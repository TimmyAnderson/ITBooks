1. Solution popisuje SPRING COMPONENTS a SPRING MANAGED BEANS.
2. SPRING FRAMEWORK PROGRAMS vytvaraju pri starte SPRING CONTAINER. Pre SPRING CONTAINER platia nasledujuce fakty.
   A. SPRING CONTAINER je DEPENDENCNY INJECTION CONTAINER, prostrednictvom, ktoreho SPRING FRAMEWORK implementuje DEPENDENCY INJECTION.
   B. SPRING CONTAINER obsahuje OBJECTS, ktore sa nazyvaju SPRING MANAGED BEANS.
   C. SPRING CONTAINER vykonava nad SPRING MANAGED BEANS nasledujuce OPERATIONS.
      A. Vytvaranie SPRING MANAGED BEANS.
      B. Manazment SPRING MANAGED BEANS.
      C. Konfiguraciu a riesenie DEPENDENCIES medzi jednotlivymi SPRING MANAGED BEANS.
   D. SPRING MANAGED BEANS, ktore su sucastou SPRING CONTAINER pouziva SPRING FRAMEWORK na implementaciu DEPENDENCY INJECTION.
   !!! E. Nie kazdy OBJECT vytvoreny v SPRING FRAMEWORK PROGRAMS musi byt sucastou SPRING CONTAINER. LOCAL OBJECTS, ako su STRINGS, ktore sa nepouzivaju v DEPENDENCY INJECTION nie su ulozene v SPRING CONTAINER.
3. SPRING CONTAINER je implementovany prostrednictvom CLASS [SpringApplication]. Pre CLASS [SpringApplication] platia nasledujuce fakty.
   A. CLASS [SpringApplication] implementuje SPRING CONTAINER.
   B. Instanciu CLASS [SpringApplication] je mozne vytvorit nasledujucimi sposobmi.
      A. Volanim CONSTRUCTOR. Instancia CLASS [SpringApplication] sa volanim CONSTRUCTOR vytvara v nasledujucich krokoch.
         A. Zavola sa CONSTRUCTOR [SpringApplication(Class<?>... primarySources)], ktory ma ako PARAMETER CLASS TYPES, ktore obsahuju CONFIGURATION SPRING CONTAINER.
         B. Vykona sa CONFIGURATION instancie CLASS [SpringApplication] volanim jej METHODS.
         C. Zavola sa METHOD [ConfigurableApplicationContext SpringApplication.run(String... args)], ktora ma ako PARAMETER ARGUMENTS zaslane do MAIN METHOD. Tato METHOD spusti SPRING CONTAINER.
      B. Pouzitim CLASS [SpringApplicationBuilder]. Instancia CLASS [SpringApplication] sa pouzitim CLASS [SpringApplicationBuilder] vytvara v nasledujucich krokoch.
         A. Vytvori sa instancia CLASS [SpringApplicationBuilder] volanim CONSTRUCTOR [SpringApplicationBuilder(Class<?>... sources)], ktory ma ako PARAMETER CLASS TYPES, ktore obsahuju CONFIGURATION SPRING CONTAINER.
         B. Vykona sa CONFIGURATION instancie CLASS [SpringApplicationBuilder] volanim jej METHODS.
         C. Zavola sa METHOD [ConfigurableApplicationContext SpringApplicationBuilder.run(String... args)], ktora ma ako PARAMETER ARGUMENTS zaslane do MAIN METHOD. Tato METHOD spusti SPRING CONTAINER.
      C. Pouzitim STATIC METHOD [ConfigurableApplicationContext SpringApplication.run(Class<?>[] primarySources, String[] args)]. STATIC METHOD ma PARAMETER [primarySources] CLASS TYPES, ktore obsahuju CONFIGURATION SPRING CONTAINER a PARAMETER [args], ktory obsahuje ARGUMENTS zaslane do MAIN METHOD. Tato METHOD vytvori instanciu CLASS [SpringApplication] a spusti SPRING CONTAINER.
   C. Pri vykonani METHOD [ConfigurableApplicationContext SpringApplication.run(String... args)] sa mozu stat nasledujuce cinnosti.
      A. Ak sa v CONFIGURATION SPRING CONTAINER nachadza ERROR, dojde k vzniku EXCEPTION.
      B. Ak je CONFIGURATION SPRING CONTAINER v poriadku, vykonaju sa nasledujuce cinnosti.
         A. METHOD [run()] spusti SPRING CONTAINER.
         B. METHOD [run()] inicializuje vsetky nakonfigurovane SPRING MANAGED BEANS.
         !!! C. METHOD [run()] sa ukonci.
         D. Riadenie sa vrati do METHOD [main()], ktora spravidla nema ziaden dalsi CODE a hned sa ukonci.
         !!!!! E. Ak SPRING CONTAINER ma v CONFIGURATION ulozene napriklad spustenie TOMCAT WEB SERVER, METHOD [run()] ho spusti spustenim separatneho THREAD, takze aj po skonceni METHOD [main()] PROGRAM dalej pokracuje.
      C. Ak je CONFIGURATION SPRING CONTAINER v poriadku, CONFIGURATION obsahuje napriklad spustenie INTERACTIVE SHELL, alebo GUI PROGRAM, vykonaju sa nasledujuce cinnosti.
         A. METHOD [run()] spusti SPRING CONTAINER.
         B. METHOD [run()] inicializuje vsetky nakonfigurovane SPRING MANAGED BEANS.
         !!! C. METHOD [run()] sa dostane do LOOP, az kym nie je INTERACTIVE SHELL, alebo GUI PROGRAM ukonceny.
         D. Riadenie sa vrati do METHOD [main()], ktora spravidla nema ziaden dalsi CODE a hned sa ukonci.
4. METHOD [ConfigurableApplicationContext SpringApplication.run(String... args)] vracia instanciu INTERFACE [ConfigurableApplicationContext]. Pre INTERFACE [ConfigurableApplicationContext] platia nasledujuce fakty.
   A. INTERFACE [ConfigurableApplicationContext] je INTERFACE, ktory reprezentuje BEAN FACTORY umoznujuci spravu SPRING MANAGED BEANS.
   B. INTERFACE [ConfigurableApplicationContext] umoznuje vkladanie INTERNATIONALIZED RESOURCE FILES.
   C. INTERFACE [ConfigurableApplicationContext] umoznuje nacitanie RESOURCES pomocou CLASS [ResourceLoader].
   D. INTERFACE [ConfigurableApplicationContext] umoznuje zasielat a prijimat EVENTS zo SPRING CONTAINER.
   E. INTERFACE [ConfigurableApplicationContext] je DERIVED z INTERFACE [ListableBeanFactory], ktory obsahuje METHOD [String[] ListableBeanFactory.getBeanDefinitionNames()]. Tato METHOD vracia zoznam vsetkych SPRING MANAGED BEANS.
   !!! E. INTERFACE [ConfigurableApplicationContext] je DERIVED z INTERFACE [ListableBeanFactory] a to je DERIVED z INTERFACE [BeanFactory]. Tento INTERFACE definuje mnozstvo METHODS pomocou ktorych je mozne ziskat instancie SPRING MANAGED BEANS.
!!!!! 5. Zasadny rozdiel medzi SPRING FRAMEWORK PROGRAMS a SPRING BOOT PROGRAMS je, ze v SPRING FRAMEWORK PROGRAMS je nutne CONFIGURATION vsetkych SPRING MANAGED BEANS nutne vykonat manualne. V SPRING BOOT PROGRAMS pri spusteni dochadza k procesu AUTOCONFIGURATION, ked su SPRING MANAGED BEANS nakonfigurovane AUTOMATICKY.

1. Solution popisuje zakladne charakteristiky SPRING BOOT LIBRARY.
2. SPRING BOOT LIBRARY je ENTERPRISE LIBRARY sluziaca na vyvoj ENTERPRISE PROGRAMS.
!!! 3. SPRING BOOT LIBRARY je alternativou k JAKARTA ENTERPRISE EDITION, co je ENTERPRISE LIBRARY vyvijana priamo ORACLE.
4. SPRING BOOT LIBRARY oproti JAKARTA ENTERPRISE EDITION ma lepsiu podporu pre MICROSERVICES a je viac LIGHTWEIGHT.
5. JAVA PLATFORM STANDARD EDITION, ktora bola vydana v roku 1995 nestaci na vyvoj ENTERPRISE PROGRAMS, pretoze nema dostatocnu podporu pre mnohe ENTERPRISE TECHNOLOGIES, ako su WEB SERVICES, DYNAMIC WEB PAGE GENERATION, DATABASE ACCESS, MESSAGING SYSTEMS, PROGRAM MONITORING ci EMAILING. Pre vyvoj ENTERPRISE PROGRAMS su nutne pokrocilejsie FRAMEWORKS ako JAVA PLATFORM STANDARD EDITION.
6. Po definovani JAVA PLATFORM STANDARD EDITION v roku 1996 SUN MICROSYSTEMS definoval SERVLET API. Pre SERVLET API platia nasledujuce fakty.
   A. SERVLET API umoznuje vyvoj WEB PROGRAMS, ktore mozu bezat vo WEB SERVERS.
   B. SERVLET API je velmi limitovane API, ktore iba umoznuje reagovat na WEB REQUESTS vo WEB SERVERS, ale neposkytuje ziadne dodatocne ENTERPRISE LIBRARIES.
7. SUN MICROSYSTEMS v roku 1999 definoval standard pre ENTERPRISE PROGRAMS, nazyvany JAVA 2 PLATFORM ENTERPRISE EDITION (J2EE). Pre tuto PLATFORM platia nasledujuce fakty.
   A. JAVA 2 PLATFORM ENTERPRISE EDITION bola specifikacia, ktora urcovala, ake funkcionality musi implementacia JAVA 2 PLATFORM ENTERPRISE EDITION splnat.
   B. JAVA 2 PLATFORM ENTERPRISE EDITION bola implementovana formou APPLICATION SERVERS, ktore boli implementovane vyznamnymi firmami ako IBM a umoznovali beh JAVA ENTERPRISE PROGRAMS.
   C. Prve verzie JAVA 2 PLATFORM ENTERPRISE EDITION APPLICATION SERVERS boli pomale a narocne na MEMORY.
   D. JAVA 2 PLATFORM ENTERPRISE EDITION nedefinovala vsetky potrebne LIBRARIES, a preto mnohe APPLICATION SERVERS definovali vlastne LIBRARIES co viedlo k neprenositelnosti JAVA ENTERPRISE PROGRAMS medzi jednotlivymi APPLICATION SERVERS.
   E. BUSSINESS LOGIC JAVA 2 PLATFORM ENTERPRISE EDITION PROGRAMS bola silne zviazane s API JAVA 2 PLATFORM ENTERPRISE EDITION co sposobovalo komplikovanu implementaciu a zlozite testovanie.
   F. JAVA 2 PLATFORM ENTERPRISE EDITION bola definovana este pred prichodom niektorych JAVA FEATURES ako su ANNOTATIONS, co znamenalo, ze DYNAMIC CONFIGURATIONS museli definovane v XML FILES.
8. Pre JAVA APPLICATION SERVERS platia nasledujuce fakty.
   A. JAVA APPLICATION SERVER je PROGRAM, ktory obsahuje WEB SERVER.
   B. JAVA APPLICATION SERVER je spusteny pri starte COMPUTER a umoznuje dynamicky pridavat a odoberat WEB PROGRAMS do JAVA APPLICATION SERVER.
   !!!!! C. JAVA APPLICATION SERVER je analogiou MICROSOFT IIS, ako centralizovaneho WEB SERVER, ktory umoznuje bez WEB PROGRAMS.
9. ORACLE definoval novy standard pre ENTERPRISE PROGRAMS, nazyvany JAKARTA ENTERPRISE EDITION (Jakarta EE). Pre tuto PLATFORM platia nasledujuce fakty.
   A. JAKARTA ENTERPRISE EDITION je nahradou za JAVA 2 PLATFORM ENTERPRISE EDITION.
   B. JAKARTA ENTERPRISE EDITION specifikacia, ktora urcuje, ake funkcionality musi implementacia JAKARTA ENTERPRISE EDITION splnat.
   !!!!! C. JAKARTA ENTERPRISE EDITION je implementovana formou APPLICATION SERVER. To znamena, ze PROGRAMS napisane pre JAKARTA ENTERPRISE EDITION musia bezat v APPLICATION SERVER.
   D. JAKARTA ENTERPRISE EDITION definuje asi 20 LIBRARIES umoznujucich vyvoj JAVA ENTERPRISE PROGRAMS.
   !!! F. Filozoficky JAKARTA ENTERPRISE EDITION je postavena tak, ze na riesenie specifickeho problemu (napriklad OBJECT RELATION MAPPER) definuje 1 LIBRARY.
!!! 10. SPRING FRAMEWORK bol release v roku 2004 a jeho cielom bolo odstranit nedostatky JAVA 2 PLATFORM ENTERPRISE EDITION. Pre SPRING FRAMEWORK platia nasledujuce fakty.
   A. SPRING FRAMEWORK je ENTERPRISE LIBRARY umoznujuca vyvoj ENTERPRISE PROGRAMS.
   !!! B. SPRING FRAMEWORK je konkurencny PROJECT voci JAVA 2 PLATFORM ENTERPRISE EDITION.
   C. SPRING FRAMEWORK poskytuje mnozstvo LIBRARIES pre implementaciu ENTERPRISE PROGRAMS.
   D. SPRING FRAMEWORK interne vyuziv mnohe LIBRARIES z JAKARTA ENTERPRISE EDITION.
   E. SPRING FRAMEWORK ma pomerne zlozitu CONFIGURATION, vyzadujuci CONFIRGURATION jednotlivych JAVA BEANS, co vedie k velmi komplikovanemu STARTUP CODE pre PROJECTS napisane v SPRING FRAMEWORK.
   !!! F. Filozoficky SPRING FRAMEWORK je postaveny tak, ze na riesenie specifickeho problemu (napriklad OBJECT RELATION MAPPER) moze definovat VIACERO LIBRARIES.
   G. SPRING FRAMEWORK podporuje AHEAD OF TIME COMPILATION, kedy cely JAVA PROGRAM je COMPILED este pred spustenim a tym padom pre beh JAVA PROGRAM nie je nutny beh JAVA VIRTUAL MACHINE.
11. SPRING BOOT z roku 2013 je nadstavba nad SPRING FRAMEWORK. Pre SPRING BOOT platia nasledujuce fakty.
   A. SPRING BOOT je nadstavba nad SPRING FRAMEWORK, ktora vyrazne zjednodusuje konfiguraciu PROGRAMS pisanych pre SPRING FRAMEWORK.
   B. SPRING BOOT umoznuje vykonat CONFIGURATION priamo v JAVA CODE volanim prislusnych METHODS.
   !!!!! C. SPRING BOOT umoznuje spustat SPRING BOOT WEB PROGRAMS v EMBEDDED SERVLET CONTAINER (TOMCAT SERVLET CONTAINER), ktory spracuvava HTTP REQUESTS, vdaka comu nie je nutne pouzivat externe EMBEDDED SERVLET CONTAINERS.
12. SPRING FRAMEWORK a SPRING BOOT maju vlastne verzionovanie.
   A. Aktualna MAJOR VERSION SPRING FRAMEWORK je VERSION [7.0].
   B. Aktualna MAJOR VERSION SPRING BOOT je VERSION [4.0].
13. Pre SPRING BOOT PROJECT platia nasledujuce fakty.
   A. SPRING BOOT PROJECT je klasicky JAVA PROJECT, ktory ma niekolko CLASSES vlozenych do CLASSPATH.
   B. Aj ked DEPENDENCIES do CLASSPATH je v SPRING BOOT PROJECT mozne vlozit manualne, idealnejsie je pouzit BUILD TOOL ako je BUILD TOOL [MAVEN], alebo BUILD TOOL [GRADLE].
   C. SPRING BOOT PROJECT je mozne vygenerovat pomocou PROJECT GENERATOR na WEB PAGE [https://start.spring.io/].
14. SPRING BOOT PROJECT vygenerovany pomocou WEB PAGE [https://start.spring.io/] ma nasledujuce STRUCTURE.
   A. DIRECTORY [src\main\java] obsahuje JAVA SOURCE CODE.
   B. DIRECTORY [src\main\resources] obsahuje JAVA RESOURCES.
   C. DIRECTORY [src\test\java] obsahuje JAVA TEST SOURCE CODE.
   D. DIRECTORY [.mvn] obsahuje konfiguraciu MAVEN WRAPPER.
   E. FILE [pom.xml] obsahuje MAVEN CONFIGURATION FILE.
   F. FILE [mvnw] obsahuje MAVEN WRAPPER CODE.
   G. FILE [mvnw.cmd] obsahuje MAVEN WRAPPER EXECUTION FILE.
   H. FILE [.gitignore] obsahuje GIT IGNORE FILE.
   I. FILE [.gitattributes] obsahuje GIT IGNORE ATTRIBUTES FILE.
   J. FILE [HELP.md] obsahuje DOCUMENTATION FILE.
15. MAVEN WRAPPER je TOOL pre ktory platia nasledujuce fakty.
   A. MAVEN WRAPPER je TOOL, ktory umoznuje BUILD a RUN MAVEN PROJECTS bez nutnosti instalacie konkretnej verzie MAVEN BUILD TOOL.
   B. MAVEN WRAPPER automaticky pred BUILD stiahne konkretnu verziu MAVEN a vykona nad nou BUILD.
   C. MAVEN WRAPPER je sucastou SPRING BOOT PROJECT, a tym padom viacero SPRING BOOT PROJECTS moze vdaka MAVEN WRAPPER pouzivat rozlicne verzie MAVEN BUILD TOOL.
   D. MAVEN WRAPPER sa nachadza v ROOT DIRECTORY SPRING BOOT PROJECT vo FILE [mvnw.cmd].
   E. COMMAND [mvnw.cmd --version] vracia VERSION MAVEN BUILD TOOL a pouziteho JAVA COMPILER.
   F. COMMAND [mvnw.cmd spring-boot:run] vykona BUILD SPRING BOOT PROJECT a nasledne ho aj spusti.
16. FILE [pom.xml] obsahuje CONFIGURATION SPRING BOOT PROJECT. FILE obsahuje nasledujuce ELEMENTS.
   A. ELEMENT [project->parent] obsahuje odkaz na PARENT POM FILE, z ktoreho su zdedene SETTINGS do CURRENT FILE [pom.xml].
   B. ELEMENT [project->groupId] obsahuje GROUP ID PROGRAMU.
   C. ELEMENT [project->artifactId] obsahuje ARTIFACT ID PROGRAMU.
   D. ELEMENT [project->version] obsahuje VERSION PROGRAMU.
   E. ELEMENT [project->properties] obsahuje CUSTOM PROPERTIES PROGRAMU.
   F. ELEMENT [project->dependencies] obsahuje DEPENDENCIES PROGRAMU.
   G. ELEMENT [project->build->plugins] obsahuje PLUGINS, ktore su potrebne pre BUILD PROGRAMU.
17. SPRING BOOT PROJECT pouziva ako PARENT POM PACKAGE [spring-boot-starter-parent]. Pre POM PACKAGE [spring-boot-starter-parent] platia nasledujuce fakty.
   A. POM PACKAGE [spring-boot-starter-parent] obsahuje v POM FILE zoznam BUILD RESOURCES, BUILD PLUGINS a PROFILES, ktore su potrebne pre BUILD SPRING BOOT PROJECT.
   B. POM PACKAGE [spring-boot-starter-parent] obsahuje v POM FILE JAVA VERSION, ktora je pouzita pre BUILD SPRING BOOT PROJECT.
   C. POM PACKAGE [spring-boot-starter-parent] ma za PARENT POM PACKAGE [spring-boot-dependencies].
18. POM PACKAGE [spring-boot-dependencies] je zakladny PACKAGE SPRING BOOT PROJECTS. Pre POM PACKAGE [spring-boot-dependencies] platia nasledujuce fakty.
   A. POM PACKAGE [spring-boot-dependencies] je zakladny PACKAGE SPRING BOOT PROJECTS, ktory obsahuje zoznam verzii jednotlivych DEPENDENCIES, ktore sa pouzivaju v SPRING BOOT PROJECTS.
   !!! B. POM PACKAGE [spring-boot-dependencies] reprezentuje BILL OF MATERIALS (BOM) FILE, ktory definuje verzie SPRING BOOT DEPENDENCIES pre ktore bola dana verzia SPRING BOOT FRAMEWORK otestovana.
   C. POM PACKAGE [spring-boot-dependencies] definuje DEFAULT VERSIONS pre jednotlive DEPENDENCIES. To umoznuje, ze POM FILES jednotlivych SPRING BOOT PROJECTS ak vlozia do PROJECT nejaky DEPENDENCIES, NEMUSIA vkladat aj jej VERSION, pretoze sa pouzije VERSION definovana v POM PACKAGE [spring-boot-dependencies].
   !!! D. POM PACKAGE [spring-boot-dependencies] NEVKLADA do SPRING BOOT PROJECT ziadnu DEPENDENCIES, ale iba nastavuje verzie jednotlivych DEPENDENCIES, pre ktore bol SPRING BOOT FRAMEWORK konrektne otestovany a je zabezpecene, ze vzajomne funguju.
   E. POM PACKAGE [spring-boot-dependencies] POM FILE je tvoreny nasledujucimi castami.
      A. V ELEMENT [project->properties] obsahuje zoznam KEY-VALUE PROPERTIES. Pre KEY VALUE PROPERTIES platia nasledujuce fakty.
         A. PROPERTY KEY je nazov ARTIFACT ID danej DEPENDENCY.
         B. PROPERTY VALUE je VERSION danej DEPENDENCY, preto ktoru bol SPRING BOOT FRAMEWORK otestovany.
         !!! C. Pouzitie KEY-VALUE PROPERTIES umoznuje jednoduchsiu zmenu VERSION danej DEPENDENCY prepisanim VALUE prislusnej KEY VALUE PROPERTY v POM FILE daneho SPRING BOOT PROJECT.
      B. V ELEMENT [project->dependencyManagement->dependencies] obsahuje zoznam vsetkych DEPENDENCIES, ktore definuje SPRING BOOT PROJECT. Pre zoznam DEPENDENCIES platia nasledujuce fakty.
         A. Kazda DEPENDENCY ma definovany GROUP ID.
         B. Kazda DEPENDENCY ma definovany ARTIFACT ID.
         !!! C. Kazda DEPENDENCY ma definovanu VERSION pomocou REFERENCE na prislusnu KEY VALUE PROPERTY definovanej v ELEMENT [project->dependencyManagement->dependency].
         !!!!! D. DEPENDENCIES v ELEMENT [project->dependencyManagement->dependencies] NIE su VLOZENE do CLASSPATH SPRING BOOT PROJECT. Su to iba referencie obsahujuce VERSIONS vsetkych DEPENDENCIES, ktore umoznuju v POM FILE SPRING BOOT PROJECT vkladat do POM FILE DEPENDENCIES bez nutnosti specifikovania konkretnej VERSION. VERSION sa pouzije z POM PACKAGE [spring-boot-dependencies].
19. Pre POM FILE ELEMENT [project->dependencyManagement] platia nasledujuce fakty.
   A. ELEMENT [project->dependencyManagement] obsahuje zoznam DEPENDENCIES, kde kazda DEPENDENCY obsahuje svoju VERSION.
   B. Zmyslom ELEMENT [project->dependencyManagement] je definovanie VERSIONS jednotlivych DEPENDENCIES, ktore sa maju pouzit, ak je DEPENDENCY vlozena do ELEMENT [spring-boot-dependencies], pricom v DEPENDENCY nie je priamo definovana VERSION danej DEPENDENCY.
   !!!!! C. DEPENDENCIES vlozene do ELEMENT [project->dependencyManagement] NIE SU vlozene do CLASSPATH daneho PROJECT. Pouzitie ELEMENT [project->dependencyManagement] iba na definovanie VERSIONS danych DEPENDENCIES. Ak sa ma dana DEPENDENCY vlozi do CLASSPATH SPRING BOOT PROJECT, musi byt vlozena do ELEMENT [project->dependencies]. ELEMENT [project->dependencyManagement] iba umoznuje VYNECHAT explicitnu specifikaciu VERSION danej DEPENDENCY v ELEMENT [project->dependencies].
   !!! D. ELEMENT [project->dependencyManagement] moze sluzit na vlozenie POM FILES, ktore obsahuju definicie DEPENDENCIES a ich VERSIONS. IMPORT sa vykonava nasledujucim sposobom.
      A. Do ELEMENT [project->dependencyManagement->dependencies] sa vlozi nova DEPENDENCY, ktora obsahuje referenciu na POM FILE, ktora obsahuju definicie DEPENDENCIES a ich VERSIONS.
      B. V DEPENDENCY musi byt nastavena VERSION v ELEMENT [version].
      C. V DEPENDENCY musi byt nastaveny ELEMENT [type] na VALUE [pom].
      D. V DEPENDENCY musi byt nastaveny ELEMENT [scope] na VALUE [import].
      !!! E. Ak sa nasledne vlozi do ELEMENT [project->dependencies] nova DEPENDENCY, ktora sa odkazuje na DEPENDENCY importovanu cez POM FILE IMPORT, tak sa pouzije VERSION definovana v danom POM FILE.
20. Pre POM FILE ELEMENT [project->dependencies] platia nasledujuce fakty.
   A. ELEMENT [project->dependencies] sluzi na specifikovanie DEPENDENCIES, ktore dany SPRING BOOT PROJECT vyuziva.
   !!! B. DEPENDENCIES vlozene do ELEMENT [project->dependencies] su vlozene do CLASSPATH SPRING BOOT PROJECT.
   !!!!! C. Ak pre konkretnu DEPENDENCY nie je definovana VERISON, tak MAVEN pozrie do ELEMENT [project->dependencyManagement] v POM FILE, alebo do ELEMENT [project->dependencyManagement] v PARENT POM FILES ci neobsahuju VERSION danej DEPENDENCY. Ak ano, tak sa pouzije tato VERSION. Ak nie, MAVEN hodi ERROR.
   !!! D. Ak pre konkretnu DEPENDENCY je definovana VERISON, tak VERSION uvedena v ELEMENT [project->dependencyManagement] POM FILE, alebo v ELEMENT [project->dependencyManagement] v PARENT POM FILES sa IGNORUJE.
21. VERSIONS SPRING BOOT DEPENDENCIES je mozne vykonat nasledujucimi sposobmi.
   A. Vlozenim DEPENDENCY so specifikovanou VERSION do ELEMENT [project->dependencies]. Tato VERSION prepise VERSION definovanu v ELEMENT [project->dependencyManagement].
   B. Zmenou VALUE KEY-VALUE PROPERTY, ktora je pouzita pre danu DEPENDENCY na inu VERSION.
22. MAVEN umoznuje vypisat zoznam DEPENDENCIES v TREE STRUCTURE.
   A. MAVEN COMMAND [mvn dependency:tree] zobrazuje DEPENDENCIES v TREE STRUCTURE.
   B. MAVEN WRAPPER COMMAND [mvnw dependency:tree] zobrazuje DEPENDENCIES v TREE STRUCTURE.
23. POM FILE umoznuje definovat zoznam REPOSITORIES z ktorych budu DEPENDENCIES importovane. Platia nasledujuce fakty.
   A. ELEMENT [project->repositories] obsahuje zoznam REPOSITORIES, z ktorych budu DOWNLOADED DEPENDENCIES.
   B. ELEMENT [project->pluginRepositories] obsahuje zoznam REPOSITORIES, z ktorych budu DOWNLOADED PUGIN DEPENDENCIES.
24. MAVEN PLUGIN [maven-compiler-plugin] vykonavajuci BUILD SPRING BOOT PROJECT moze byt rozsireny o ANNOTATION PROCESSORS. Pre ANNOTATION PROCESSORS platia nasledujuce fakty.
   A. ANNOTATION PROCESSORS je CUSTOM COMPONENT MAVEN PLUGIN [maven-compiler-plugin], ktory pocas BUILD SPRING BOOT PROJECT interpretuje JAVA ANNOTATIONS. JAVA ANNOTATIONS mozu byt interpretovane generovanim dodatocneho JAVA CODE, alebo vykonanim specifickych ACTIONS nad na zaklade tychto JAVA ANNOTATIONS.ANNOTATION PROCESSORS mozu z JAVA ANNOTATIONS napriklad generovat JSON FILES, obsahujuce EXTERNAL CONFIGURATIONS.
   B. ANNOTATION PROCESSORS sa vkladaju do ELEMENT [project->build->plugins->plugin->configuration->annotationProcessorPaths].
25. SPRING BOOT definuje koncept DEPENDENCY STARTERS. Pre DEPENDENCY STARTERS platia nasledujuce fakty.
   A. DEPENDENCY STARTER je DEPENDENCY, ktora obsahuje vzajomne prepojene DEPENDENCIES s nastavenymi VERSIONS, ktore boli otestovane, ze spolu funguju.
   B. DEPENDENCY STARTERS, ktore su sucastou SPRING FRAMEWORK maju v NAME PREFIX [spring-boot-starter].
   C. DEPENDENCY STARTERS, ktore NIE su sucastou SPRING FRAMEWORK by mali mat v NAME POSFTIX [spring-boot-starter].
   D. Viacere DEPENDENCY STARTERS mozu byt vzajomne kombinovane.
26. SPRING BOOT PROGRAMS nacitavaju CONFIGURATION PROPERTIES z roznych SOURCES.
   A. FILE [src/main/resources/application.properties] obsahuje CONFIGURATION PROPERTIES vo FORMATE [properties].
   B. FILE [src/main/resources/application.yml] obsahuje CONFIGURATION PROPERTIES vo FORMATE [yml].
   C. COMMAND LINE PARAMETERS.
27. Zobrazenie BANNER sa da nastavit pomocou CONFIGURATION PROPERTY [spring.main.banner-mode], ktora moze mat nasledujuce VALUES.
   A. VALUE [off]. BANNER sa vobec nezobrazi.
   B. VALUE [console]. BANNER sa zapise do CONSOLE.
   C. VALUE [log]. BANNER sa zapise CURRENT LOG STREAM.
28. SPRING BOOT PROGRAMS vykonavaju LOGGING nasledujucim sposobom.
   A. Zapis LOG RECORDS do LOGS je v SPRING FRAMEWORK mozne vykonat nasledujucimi sposobmi.
      A. Priamym pouzitim konkretnej LOG LIBRARY.
      B. Pouzitim niektorej z implementovanych LOG FACADES nad LOG LIBRARIES.
   B. Pouzitie FACADES pre LOG RECORDS do LOGS je preferovane, pretoze FACADE abstrahuje od implementacnych detailov konkretnej LOG LIBRARY a definuje univerzalne API na zapis LOG RECORDS do LOGS.
   C. DEPENDENCY [spring-boot-starter] ma DEPENDENCY na DEPENDENCY [spring-boot-starter-logging.], ktora definuje nasledujuce LOG FACADES.
      A. LOG FACADE [SIMPLE LOGGING FACADE FOR JAVA (SLF4J)]. Toto je odporucana LOG FACASE.
      B. LOG FACADE [COMMONS LOGGING API]. Tuto LOG FACASE interne pouziva SPRING FRAMEWORK.
29. Zapis LOG RECORDS do LOGS s pouzitim LOG FACADE [SIMPLE LOGGING FACADE FOR JAVA (SLF4J)] sa vykonava v nasledujucich krokoch.
   A. Do CLASS sa vlozi FIELD TYPE [Logger].
   B. V CONSTRUCTOR sa vytvori instancia FIELD TYPE [Logger].
   C. Na zapis LOG RECORDS do LOGS sa pouzivaju METHODS FIELD TYPE [Logger].
30. Ci LOG RECORD bude vlozeny do LOG urcuje LOG CATEGORY a LOG LEVEL.
   A. LOG CATEGORY je STRING VALUE, ktora sa zastav pri vytvarani instancie CLASS [Logger].
   B. LOG LEVEL urcuje zavaznost daneho LOG RECORD. Do LOGS su zapisane tie LOG RECORDS, ktorych zavaznost sa rovna, alebo je vyssia ako nastavena uroven. LOG FACADE [SIMPLE LOGGING FACADE FOR JAVA (SLF4J)] definuje nasledujuce LOG LEVELS.
      A. LOG LEVEL [OFF]. LOG RECORD nebude zapisany do LOG.
      B. LOG LEVEL [TRACE].
      C. LOG LEVEL [DEBUG].
      D. LOG LEVEL [INFO]. Toto je DEFAULT LOG LEVEL.
      E. LOG LEVEL [WARN].
      F. LOG LEVEL [ERROR].
      G. LOG LEVEL [FATAL].
31. Pre nastavenie LOG LEVELS platia nasledujuce fakty.
   A. LOG LEVELS sa nastavuju v CONFIGURATION FILES.
   B. LOG LEVEL SETTING ma FORMAT [logging.level.NAMESPACE.CLASS_NAME].
      A. VALUE [logging] je povinna VALUE.
      B. VALUE [level] je povinna VALUE.
      C. VALUE [NAMESPACE] obsahuje NAMESPACE na ktory sa ma LOG LEVEL aplikovat.
      D. VALUE [CLASS_NAME] je OPTIONAL CLASS NAME na ktory sa ma LOG LEVEL aplikovat.
    C. Ak je LOG LEVEL aplikovany na neuplny NAMESPACE, aplikuje sa na vsetky CHILD NAMESPACES.
    D. LOG LEVEL nastaveny pre CLASS prepisuje nastavenia na NAMESPACES.
    E. LOG LEVEL je mozne nastavit aj na SPRING BOOT CLASSES.

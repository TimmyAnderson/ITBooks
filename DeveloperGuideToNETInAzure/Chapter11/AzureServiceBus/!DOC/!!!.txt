1. SOLUTION demonstruje pouzitie AZURE SERVICE BUS v .NET CORE.
2. Pre AZURE SERVICE BUS platia nasledujuce fakty.
   A. AZURE SERVICE BUS je MESSAGING technologia, umoznujuca zasielat MESSAGES medzi SERVICES.
   B. AZURE SERVICE BUS implementuje MESSAGE QUEUES do ktorych su MESSAGES ukladane FIRST IN FIRST OUT sposobom.
   !!! C. V AZURE SERVICE BUS je kazda MESSAGE citana vzdy iba 1 RECEPIENT, pricom po jej vycitani je MESSAGE odstranena z MESSAGE QUEUE.
   !!! D. AZURE SERVICE BUS MESSAGE QUEUES podporuju TRANSACTIONS. To znamena, ze odosielanie a citanie MESSAGES z MESSAGE QUEUES moze byt sucastou TRANSACTION, ktora moze zahrnat aj ine TRANSACTIONAL RESOURCES.
   E. AZURE SERVICE BUS podporuje BATCH SENDING a BATCH RECEIVING viacerych MESSAGES z MESSAGE QUEUES.
   F. AZURE SERVICE BUS umoznuje AUTO FORWARDING MESSAGES z jednej MESSAGE QUEUE do inej MESSAGE QUEUE.
   G. AZURE SERVICE BUS podporuje SCHEDULER DELIVERY, kedy MESSAGES su nie su odoslane do MESSAGE QUEUES okamzite, ale az v definovanom case.
   H. AZURE SERVICE BUS podporuje DEAD LETTER QUEUES do ktorych su ukladane MESSAGES, ktore nemohli byt dorucene do MESSAGE QUEUES, alebo spracovane z MESSAGE QUEUES.
   I. AZURE SERVICE BUS podporuje ADVANCED MESSAGE QUEUEING PROTOCOL (AMQP).
3. AZURE SERVICE BUS funguje nasledujucim sposobom.
   A. SENDER (PUBLISHER) posiela MESSAGES do definovanych TOPICS.
   B. RECEIVERS (SUBSCRIBERS) citaju tieto MESSAGES z definovanych SUBSCRIPTIONS.
4. COMMUNICATION prostrednictvom AZURE SERVICE BUS podporuje nasledujuce COMMUNICATION PATTERNS.
   A. AZURE SERVICE BUS umoznuje, aby PUBLISHERS a SUBSCRIBERS boli od seba logicky uplne oddeleni. To znamena, ze PUBLISHERS nevedia nic o SUBSCRIBERS a SUBSCRIBERS nevedia nic o PUBLISHERS.
   B. AZURE SERVICE BUS podporuje PATTERN LOAD LEVELING, kedy PUBLISHERS aj SUBSCRIBERS mozu odosielat a citat MESSAGES svojim vlastnym tempom. Nespracovane MESSAGES ostavaju v QUEUE, az kym ich SUBSCRIBERS nespracuju.
   !!! C. AZURE SERVICE BUS podporuje PATTERN COMPETING COSNUMER, kedy AZURE v pripade, ze v MESSAGE QUEUE je prilis vela MESSAGES, pretoze ich SUBSCRIBER nestiha spracovavat, moze dynamicky vytvorit novych SUBSCRIBERS, ktori budu paralelne spracovavat viacero MESSAGES z MESSAGE QUEUE v danom case. No stale plati, ze kazdy SUBSCRIBER spracovava iba svoj SET MESSAGES a nedochadza k duplicitnemu spracovanie tej istej MESSAGE viacerymi SUBSCRIBERS.
5. TOPICS a SUBSCRIPTIONS umoznuju aby 1 MESSAGE bola spracovana viacerymi SUBSCRIBERS. Platia nasledujuce fakty.
   A. Pri priamom pouziti MESSAGE QUEUES je 1 MESSAGE odoslana PUBLISHER do MESSAGE QUEUE, odkial ju vycitava vzdy 1 SUBSCRIBER.
   B. Pri pouziti TOPICS a SUBSCRIPTIONS moze byt 1 MESSAGE odoslana do MESSAGE QUEUE spracovana viacerymi SUBSCRIBERS.
   C. TOPICS a SUBSCRIPTIONS funguju nasledujucim sposobom.
      A. PUBLISHER neodosiela MESSAGES priamo do MESSAGE QUEUE, ale odosiela ich do TOPICS, ktore su nadstavbou nad MESSAGE QUEUE.
      B. Kazda MESSAGE, ktora bola PUBLISHED do MESSAGE QUEUE nie je vycitavana priamo z MESSAGE QUEUES, ale zo SUBSCRIPTIONS, ktore su nadstavbou nad MESSAGE QUEUE.
      C. SUBSCRIPTIONS su pre jednotlive TOPICS REGISTROVANE v AZURE SERVICE BUS.
      !!! D. Kazda SUBSCRIPTION umoznuje spracovanie MESSAGES z MESSAGE QUEUES, ktore je NEZAVISLE od ostatnych SUBSCRIPTIONS. To znamena, ze jedna a ta ista MESSAGE moze byt spracovana viacerymi SUBSCRIBERS, ak su pripojene k rozlicnym SUBSCRIPTIONS.
      !!! E. Distribucia MESSAGES medzi TOPICS a SUBSCRIPTIONS je dana nastavenim RULES. BY-DEFAULT, kazda MESSAGE, ktora dojde do TOPIC je distribuovana do vsetkych registrovanych SUBSCRIPTIONS. No SUBSCRIPTIONS mozu mat nastavene RULES, ktore filtruju, ake MESSAGES maju byt zasielane do danych SUBSCRIPTIONS. SUBSCRIPTIONS zaroven mozu mat nastavene ACTIONS, ktore mozu modifikovat dane MESSAGES predtym, ako su odoslane do prislusnej SUBSCRIPTION.
6. AZURE SERVICE BUS umoznuje nacitavat MESSAGES nasledujucimi sposobmi.
   A. MODE [RECEIVE AND DELETE]. Pri tomto MODE je MESSAGE pri uspesnom nacitani z MESSAGE QUEUE z MESSAGE QUEUE odstranena. Nevyhodou tohto MODE je, ze ak po vycitani MESSAGE z MESSAGE QUEUE dojde k ERROR v SUBSCRIBER a ta nedokonci jej spracovanie, MESSAGE je navzdy stratena.
   B. MODE [PEEK LOCK]. Pri tomto MODE prebieha spracovanie MESSAGE nasledujucim sposobom.
      A. MESSAGE po vycitavani SUBSCRIBER z MESSAGE QUEUE je LOCKED na urceny TIMEOUT.
      B. SUBSCRIBER po uspesnom spracovani oznaci MESSAGE za CONSUMED a MESSAGE je nasledne odstranena z MESSAGE QUEUE.
      !!! C. Ak SUBSCRIBER nedokaze MESSAGE spracovat, oznaci MESSAGE za ABANDONED a MESSAGE QUEUE uvolni LOCK nad danou MESSAGE, aby mohla byt spracovana inym SUBSCRIBER.
      !!! D. Ak SUBSCRIBER do stanoveneho TIMEOUT neoznaci MESSAGE sa CONSUMED, ani ABANDONED, tak MESSAGE QUEUE uvolni LOCK nad danou MESSAGE, aby mohla byt spracovana inym SUBSCRIBER.
7. AZURE SERVICE BUS podporuje PARTITIONING MESSAGE QUEUES. Pre PARTITIONING platia nasledujuce fakty.
   A. NON-PARTITIONED MESSAGE QUEUES maju iba 1 MESSAGE BROKER, ktory vycitava MESSAGES z MESSAGE QUEUE a 1 MESSAGE QUEUE STORAGE, kde su MESSAGES ukladane.
   B. PARTITIONED MESSAGE QUEUES maju viacero PARTITIONS, kde pre kazdu PARTITION existuje nezavisly MESSAGE BROKER a nezavisly MESSAGE QUEUE STORAGE.
   C. PARTITIONED MESSAGE QUEUES umoznuju existenciu viacerych nezavislych SUBSCRIBERS, ktore citaju MESSAGES z tej istej MESSAGE QUEUE a tym zvysuju rychlost spracovanie MESSAGES.
   D. PARTITIONED MESSAGE QUEUES umoznuju pri ERROR v jednej PARTITION pokracovat vo vycitavani MESSAGES z inych MESSAGE QUEUE PARTITIONS.
   !!!!!! E. PARTITIONED MESSAGE QUEUES NEGARANTUJU FIRST IN FIRST OUT MESSAGE PROCESSING. Pri PARTITIONED MESSAGE QUEUES dostane SUBSCRIBER najstarsiu MESSAGE tej PARTITION, ktore ako prva odpovie na poziadavku na zaslanie MESSAGE. Tato MESSAGE vsak nemusi byt najstarsou MESSAGE v celej MESSAGE QUEUE.
   F. Kazda PARTITION ma vlastny PARTITION KEY.
   G. PARTITIONED MESSAGE QUEUES funguju nasledujucim sposobom.
      A. Po odoslani MESSAGE PUBLISHER do PARTITIONED MESSAGE QUEUE AZURE SERVICE BUS odosle MESSAGE do jeden z vytvorenych PARTITIONS, bud nahodne, alebo na zaklade definovaneho PARTITION KEY.
      !!! B. Ked SUBSCRIBER sa pokusi vycitat MESSAGE z PARTITIONED MESSAGE QUEUE, tak AZURE SERVICE BUS oslovi vsetky PARTITIONS, aby vratili svoje najstarsie MESSAGES. Do SUBSCRIBER je zaslana MESSAGE z tej PARTITION, ktora ako PRVA odpovedala na poziadavku na zaslanie svojich najstarsich MESSAGES. To znamena, ze vratena MESSAGE NEMUSI byt nutne NAJSTARSOU MESSAGE v celej PARTITIONED MESSAGE QUEUE.
      C. SERVICE BUS si uklada do CACHE vratene najstarsie MESSAGES z jednotlivych PARTITIONS, aby pri dalsom citani mohol vratit tieto MESSAGES z CACHE a nemusel znova oslovovat jednotlive PARTITIONS.
8. AZURE SERVICE BUS podporuje NAMESPACES. Pre NAMESPACES platia nasledujuce fakty.
   A. NAMESPACE je CONTAINER, ktory obsahuje vsetky MESSAGING COMPONENTS, ako su MESSAGE QUEUES, alebo TOPICS.
   B. NAMESPACE moze mat jednu, alebo viacero MESSAGE QUEUES a TOPICS.
   C. NAMESPACES su automaticky distribuovane na 3 AZURE AVAILABILITY ZONES.
9. AZURE SERVICE BUS podporuje nasledujuce SECURITY MECHANISMS.
   A. AZURE SERVICE BUS umoznuje AUTHENTICATION a AUTHORIZATION nasledujucimi sposobmi.
      A. Vyuzitim AZURE ACTIVE DIRECTORY s ktorou je AZURE SERVICE BUS integrovana.
      B. Pouzitim SHARED ACCESS SIGNANTURES.
   B. AZURE SERVICE BUS umoznuje definovat obmedzenia na IP ADDRESSES, ktore mozu pristupovat k MESSAGE QUEUES.
   C. AZURE SERVICE BUS podporuje ENCRYPTION MESSAGES nasledujucimi sposobmi.
      A. ENCRYPTION DATA IN TRANSIT. MESSAGES prenasane medzi PUBLISHERS, MESSAGES QUEUES su prenasane pomocou PROTOCOL TRANSPORT SECURITY LAYER (TLS), ktory umoznuje bezpecnu komunikaciu po sieti kryptovanim MESSAGES zasielanych sietou.
      B. ENCRYPTION DATA AT REST. MESSAGES ulozene v MESSAGE QUEUES su automaticky ENCRYPTED. CUSTOMER mozu definovat vlastne KEYS na ENCRYPTION MESSAGES.
10. AZURE SERVICE BUS podporuje MOTIRING MESSAGE QUEUES pomocou AZURE MONITOR.
11. AZURE SERVICE BUS podporuje DEAD LETTER QUEUES. Pre DEAD LETTER QUEUES platia nasledujuce fakty.
   A. Do DEAD LETTER QUEUES su ukladane MESSAGES, ktore nemohli byt dorucene, alebo spracovane ani po opakovanych pokusoch.
   B. DEAD LETTER QUEUES umoznuju vykonat RETRY FAILED MESSAGES, ak princiny ERRORS, ktore poslali MESSAGES do DEAD LETTER QUEUES boli odstranene.
   !!! C. USERS AZURE SERVICE BUS musia pravidelne monitorovat obsah DEAD LETTER QUEUES a vyberat z nich nespracovane MESSAGES, aby nedoslo k zaplnaniu DEAD LETTER QUEUES.
12. AZURE SERVICE BUS vyzaduje vytvorenie nasledujucich RESOURCES.
   A. RESOURCE [SERVICE BUS NAMESPACE].
   B. RESOURCE [SERVICE BUS QUEUE].
   C. RESOURCE [SERVICE BUS TOPIC].
   D. RESOURCE [SERVICE BUS SUBSCRIPTION].
13. S SERVICE BUS NAMESPACES sa pracuje pomocou nasledujucich COMMANDS.
   A. SERVICE BUS NAMESPACE sa vytvara pomocou COMMAND [az servicebus namespace create]. Prikladom je COMMAND [az servicebus namespace create --name MyServiceBusNamespaceTA76 --resource-group Chapter11 --location centralus --sku Standard --min-tls 1.2].
   B. SERVICE BUS NAMESPACE sa odstranuje pomocou COMMAND [az servicebus namespace delete]. Prikladom je COMMAND [az servicebus namespace delete --name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   C. Zoznam SERVICE BUS NAMESPACES vracia COMMAND [az servicebus namespace list]. Prikladom je COMMAND [az servicebus namespace list --resource-group Chapter11].
14. S SERVICE BUS QUEUES sa pracuje pomocou nasledujucich COMMANDS.
   A. SERVICE BUS QUEUE sa vytvara pomocou COMMAND [az servicebus queue create]. Prikladom je COMMAND [az servicebus queue create --name MyQueueTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   B. SERVICE BUS QUEUE sa odstranuje pomocou COMMAND [az servicebus queue delete]. Prikladom je COMMAND [az servicebus queue delete --name MyQueueTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   C. Zoznam SERVICE BUS QUEUES vracia COMMAND [az servicebus queue list]. Prikladom je COMMAND [az servicebus queue list --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
15. S SERVICE BUS TOPICS sa pracuje pomocou nasledujucich COMMANDS.
   A. SERVICE BUS TOPIC sa vytvara pomocou COMMAND [az servicebus topic create]. Prikladom je COMMAND [az servicebus topic create --name MyTopicTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   B. SERVICE BUS TOPIC sa odstranuje pomocou COMMAND [az servicebus topic delete]. Prikladom je COMMAND [az servicebus topic delete --name MyTopicTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   C. Zoznam SERVICE BUS TOPICS vracia COMMAND [az servicebus topic list]. Prikladom je COMMAND [az servicebus topic list --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
16. S SERVICE BUS SUBSCRIPTIONS sa pracuje pomocou nasledujucich COMMANDS.
   A. SERVICE BUS SUBSCRIPTION sa vytvara pomocou COMMAND [az servicebus topic subscription create]. Prikladom je COMMAND [az servicebus topic subscription create --name MySubscriptionTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11 --topic-name MyTopicTA76 --max-delivery-count 10 --enable-dead-lettering-on-message-expiration true --status Active].
   B. SERVICE BUS SUBSCRIPTION sa odstranuje pomocou COMMAND [az servicebus topic subscription delete]. Prikladom je COMMAND [az servicebus topic subscription delete --name MySubscriptionTA76 --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11 --topic-name MyTopicTA76].
   C. Zoznam SERVICE BUS SUBSCRIPTIONS vracia COMMAND [az servicebus topic subscription list]. Prikladom je COMMAND [az servicebus topic subscription list --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11 --topic-name MyTopicTA76].
17. Na pristup k AZURE SERVICE BUS je potrebny CONNECTION STRING na SERVICE BUS NAMESPACE. DEFAULT CONNECTION STRING na SERVICE BUS NAMESPACE sa ziskava v nasledujucich krokoch.
   A. Pomocou COMMAND [az servicebus namespace authorization-rule list] sa ziska zoznam AUTHORIZATION RULES pre dany SERVICE BUS NAMESPACE. Prikladom je COMMAND [az servicebus namespace authorization-rule list --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].
   B. BY-DEFAULT pre SERVICE BUS NAMESPACE sa vytvara AUTHORIZATION RULE [RootManageSharedAccessKey], ktora ma ACCESS RIGHTS na pristup k AZURE SERVICE BUS.
   C. Pomocou COMMAND [az servicebus namespace authorization-rule keys list] sa ziska CONNECTION STRING na SERVICE BUS NAMESPACE. Prikladom je COMMAND [az servicebus namespace authorization-rule keys list --authorization-rule-name RootManageSharedAccessKey --namespace-name MyServiceBusNamespaceTA76 --resource-group Chapter11].

40. AZURE SERVICE BUS umoznuje pre SERVICE BUS SUBSCRIPTIONS definovat FILTERS. Pre FILTERS platia nasledujuce fakty.
   A. SERIVCE BUS SUBSCRIPTIONS umoznuju existenciu paralelnych SUBSCRIBERS, ktori citaju MESSAGES z danej SERIVCE BUS SUBSCRIPTION, ktora prijima MESSAGE z SERIVCE BUS TOPIC.
   B. FILTERS umoznuju filtrovat, ktore MESSAGES z SERIVCE BUS TOPIC budu odoslane do danej SERIVCE BUS SUBSCRIPTION.
   C. FILTERS su definovane pomocou FILTER RULES. FILTER RULES obsahuju nasledujuce ELEMENTS.
      A. FILTER CONDITION. Obsahuje CONDITION daneho FILER.
      B. FILTER ACTION. Obsahuje ACTION daneho FILTER. ACTION je OPTIONAL a moze menit PROPERTIES (CREATE, UPDATE, DELETE) danej MESSAGE.
   !!! D. Vsetky FILTER RULES, ktore NEMAJU definovanu FILTER ACTION su kombinovane pomocou OPERATOR OR. To znamena, ze ak MESSAGE splna 1, alebo viacero FILTER RULES, ktore NEMAJU definovanu FILTER ACTION, tak do SERIVCE BUS SUBSCRIPTION je zaslana 1 kopia tejto MESSAGE.
   !!! E. Kazdy FILTER RULE, ktory MA definovanu FILTER ACTION, ak splna FILTER CONDITION, vygeneruje novu kopiu MESSAGE, ktora je modifikovana pomocou FILTER ACTION, a zaroven ma pridanu PROPERTY [RuleName], ktora obsahuje nazov FILTER RULE, ktora bola aplikovana na MESSAGE. To znamena, ze ak existuje 'N' FILTER RULES, ktore MAJU definovanu FILTER ACTION a splnaju FILTER CONDITION, do SERIVCE BUS SUBSCRIPTION je dorucenych 'N' MESSAGES, kde kazda MESSAGE bola modifikovana prislusnou FILTER ACTION.
   F. SERIVCE BUS SUBSCRIPTION ma po vytvoreny definovanu FILTER RULE, ktora vsetky MESSAGES zo SERIVCE BUS TOPIC preposiela do danej SERIVCE BUS SUBSCRIPTION.
   G. AZURE SERVICE BUS definuje nasledujuce typy FILTERS.
      A. SQL FILTERS.
      B. CORRELATION FILTERS.
      C. BOOLEAN FILTERS.
   !!! H. FILTERS je mozne aplikovat na APPLICATION PROPERTIES a SYSTEM PROPERTIES, ktore su spolu s MESSAGE posielane ako MESSAGE METADATA. FILTERS vsak NIE JE mozne aplikovat na MESSAGE BODY.
41. Pre SQL FILTERS platia nasledujuce fakty.
   A. SQL FILTER je reprezentovany CLASS [SqlRuleFilter].
   B. SQL FILTER umoznuje definovat CUSTOM EXPRESSIONS, ktore maju podobu EXPRESSIONS v SQL QUERIES.
   C. SQL FILTER umoznuju pouzivat okrem OPERATOR EQUAL, aj dalsie typy SQL OPERATORS. SQL FILTER zaroven rozlisuje PROPERTY TYPE ci je to STRING VALUE, INTEGER NUMBER VALUE, FLOATING POINT NUMBER VALUE, alebo BOOLEAN VALUE.
   !!! D. SQL FILTER sa aplikuje na APPLICATION PROPERTIES, ktore je mozne v SERVICE BUS MESSAGE nastavit pomocou PROPERTY [IDictionary<string,object> ServiceBusMessage.ApplicationProperties].
   !!! E. SQL FILTER je mozne aplikovat aj na SYSTEM PROPERTIES pomocou PREFIX [sys].
   F. SQL FILTER je POMALSI, ako CORRELATION FILTER.
42. Pre CORRELATION FILTERS platia nasledujuce fakty.
   A. CORRELATION FILTER je reprezentovany CLASS [CorrelationRuleFilter].
   B. CORRELATION FILTER umoznuje definovat CUSTOM EXPRESSIONS, ktore testuju ci PROPERTY ma nastaveny danu VALUE.
   !!! C. CORRELATION FILTER NEUMOZNUJE pouzitie inych OPERATORS ako OPERATOR EQUAL a NEROZLISUJE TYPE danej PROPERTY. Vsetky PROPERTIES povazuje za STRING VALUE.
   !!! D. CORRELATION FILTER sa aplikuje na APPLICATION PROPERTIES, ktore je mozne v SERVICE BUS MESSAGE nastavit pomocou PROPERTY [IDictionary<string,object> ServiceBusMessage.ApplicationProperties].
   !!! E. CORRELATION FILTER je mozne aplikovat aj na SYSTEM PROPERTIES.
   F. CORRELATION FILTER je RYCHLEJSI, ako SQL FILTER.
43. Pre BOOLEAN FILTERS platia nasledujuce fakty.
   A. BOOLEAN FILTER je reprezentovany CLASS [TrueFilter], alebo CLASS [FalseFilter ].
   B. BOOLEAN FILTER umoznuje akceptovat kazdu MESSAGE, alebo odmietnut vsetky MESSAGES.
44. Pre FILTER ACTIONS platia nasledujuce fakty.
   A. FILTER ACTIONS mozu modifikovat MESSAGES predtym, nez su odoslane do SERVICE BUS SUBSCRIPTION.
   B. FILTER ACTIONS mozu modifikovat APPLICATION PROPERTIES pristupne pomocou PROPERTY [IDictionary<string,object> ServiceBusMessage.ApplicationProperties].
   !!! C. FILTER ACTIONS mozu modifikovat aj SYSTEM PROPERTIES. No zmena nie kazdej SYSTEM PROPERTIY sposobi zmenu chovania sa MESSAGE. Niektore SYSTEM PROPERTIES su spracovane uz pri nacitani MESSAGE a ich modifikacia nema vplyv na chovanie sa MESSAGE.
   D. FILTER ACTIONS mozu modifikovat aj naraz viacero APPLICATION PROPERTIES a SYSTEM PROPERTIES, ktore su oddelene pomocou CHARACTER [;].
   E. Pridanie, alebo modifikacia PROPERTIES sa vykonava pomocou SYNTAX [SET PROPERTY_NAME=EXPRESSION], kde VALUE [PROPERTY_NAME] je nazov PROPERTY a VALUE [EXPRESSION] je EXPRESSION, na ktoru sa ma nastavit PROPERTY VALUE.
   F. Odstranenie existujucich PROPERTIES sa vykonava pomocou SYNTAX [REMOVE PROPERTY_NAME], kde VALUE [PROPERTY_NAME] je nazov PROPERTY.
   !!!!! G. Podla dokumentacia je FILTER ACTIONS mozne aplikovat iba na SQL FILTERS. No na CORRELATION FILTER sa FILTER ACTIONS tiez aplikovali, ale IBA pre pridanie a modifikaciu novej PROPERTY. Odstranenie existujucej PROPERTY NEFUNGOVALO.

54. AZURE SERVICE BUS podporuje RETRIES pri odosielani a citani MESSAGES. Pre MESSAGE RETRIES platia nasledujuce fakty.
   A. MESSAGE RETRIES podporuje nasledujuce MODES.
      A. EXPONENTIAL MESSAGE RETRIES MODE. Pri tomto MODE sa cas dalsieho RETRY exponencialne zvysuje.
      B. FIXED MESSAGE RETRIES MODE. Pri tomto MODE je cas dalsieho RETRY konstatny.
      C. CUSTOM MESSAGE RETRIES MODE. Pri tomto MODE cas dalsieho RETRY urcuje implementacia CUSTOM CLASS, ktora je DERIVED z CLASS [ServiceBusRetryPolicy].
   B. MESSAGE RETRIES PROPERTIES su definovane pomocou instancie CLASS [ServiceBusRetryOptions].
   C. MESSAGE RETRIES sa nastavuje pre CLASS [ServiceBusClient] pomocou CLASS [ServiceBusClientOptions] a jej PROPERTY [ServiceBusRetryOptions ServiceBusClientOptions.RetryOptions].
55. CLASS [ServiceBusRetryOptions] sluzi nastavenie RETRY SETTINGS. Pre CLASS platia nasledujuce fakty.
   A. PROPERTY [ServiceBusRetryMode Mode] urcuje RETRY MODE. Moze obsahovat VALUE [ServiceBusRetryMode.Exponential], alebo VALUE [ServiceBusRetryMode.Fixed].
   B. PROPERTY [int MaxRetries] urcuje maximalny pocet RETRIES, ktore sa maju vykonat. Ak OPEREATION neskonci uspesne ani po poslednom RETRY, je povazovana za zlyhanu.
   C. PROPERTY [TimeSpan Delay] urcuje DELAY medzi jednotlivymi RETRIES. Pre MODE [ServiceBusRetryMode.Fixed] sa tento cas nemeni. Pre MODE [ServiceBusRetryMode.Exponential] tato VALUE sluzi ako zaklad pre prepocet exponencialneho narastu casu.
   D. PROPERTY [TimeSpan MaxDelay] urcuje maximalny DELAY medzi jednotlivymi RETRIES. Pouziva sa pre MODE [ServiceBusRetryMode.Exponential].
   E. PROPERTY [TimeSpan TryTimeout] urcuje maximalny cas na vykonanie danej OPERATION.
   F. PROPERTY [ServiceBusRetryPolicy CustomRetryPolicy] umoznuje definovat CUSTOM RETRY POLICY definovanim vlastnej instancie CLASS [ServiceBusRetryPolicy]. Ak je tato PROPERTY nastavena, VALUES ostatnych PROPERTIES su ignorovane.
56. CLASS [ServiceBusRetryPolicy] sluzi na definovanie CUSTOM RETRY POLICY. Pre CLASS platia nasledujuce fakty.
   A. ABSTRACT METHOD [TimeSpan CalculateTryTimeout(int AttemptCount)] urcuje cas, ktory ma SERVICE BUS BROKER cakat na vykonanie OPERATION pre dany ATTEMPT.
   B. ABSTRACT METHOD [TimeSpan? CalculateRetryDelay(Exception LastException, int AttemptCount)] urcuje cas, ktory ma SERVICE BUS BROKER cakat na vykonanie dalsieho RETRY pre dany ATTEMPT. Ak METHOD vrati VALUE [null] tak k dalsiemu RETRY nedojde a OPERATION je povazovana za zlyhanu.
57. AZURE SERVICE BUS podporuje nasledujuce komunikacne PROTOCOLS.
   A. PROTOCOL [ADVANCED MESSAGE QUEUEING PROTOCOL] je standardny PROTOCOL pre pristup k MESSAGE QUEUES. Ma mnozstvo FEATURES.
   B. PROTOCOL [SERVICE BUS MESSAGING PROTOCOL] je specialne optimalizovany QUEUEING PROTOCOL pre .NET.
58. AZURE SERVICE BUS podporuje nasledujuce RECEIVE MODES.
   A. MODE [RECEIVE AND DELETE]. Tento MODE automaticky pri vybere MESSAGE z MESSAGE QUEUE ju z MESSAGE QUEUE odstrani. Nevyhodou je, ze MODE nepodporuje TRANSACTIONS. To znamena, ze ak pri spracovani MESSAGE dojde k ERROR, MESSAGE je nenavratne stratena.
   B. MODE [PEEK LOCK]. Pri tomto MODE je pri vybere MESSAGE z MESSAGE QUEUE MESSAGE LOCKED, ale nie je DELETED z MESSAGE QUEUE. MESSAGE je DELETED z MESSAGE QUEUE az ked PROGRAM spracovajuci danu MESSAGE to explicitne prikaze volanim prislusnej METHOD. Tento MODE podporuje TRANSACTIONS.
59. AZURE SERVICE BUS podporuje PREFETCHING. Pre PREFETCHING platia nasledujuce fakty.
   A. PREFETCHING umoznuje SUBSCRIBERS pri jednej komunikacii nacitat viacero MESSAGES z MESSAGE QUEUE.
   B. PREFETCHING optimalizuje PERFORMANCE, pretoze PREFETCHED MESSAGES su ulozene v LOCAL QUEUE a pri dalsej poziadavke na nacitanie MESSAGE z MESSAGE QUEUE je tato MESSAGE nacitata z CACHE, bez nutnosti vykonat komunikaciu s MESSAGE QUEUE.
   C. AZURE SERVICE BUS umoznuje definovat pocet PREFETCHED MESSAGES pomocou PROPERTY [int ServiceBusReceiverOptions.PrefetchCount].

AZURE CLI COMMANDS - SERVICE BUS NAMESPACES.

1. COMMAND [az servicebus namespace create --name NAMESPACE_NAME --resource-group RESOURCE_GROUP_NAME --location LOCATION_NAME --sku SKU_TYPE --min-tls TLS_VERSION].
   A. COMMAND vytvara novy SERVICE BUS NAMESPACE.
   B. OPTION [--name NAMESPACE_NAME] vo VALUE [NAMESPACE_NAME] obsahuje nazov SERVICE BUS NAMESPACE.
   C. OPTION [--resource-group RESOURCE_GROUP_NAME] vo VALUE [RESOURCE_GROUP_NAME] urcuje RESOURCE GROUP NAME.
   D. OPTION [--location LOCATION_NAME] vo VALUE [LOCATION_NAME] urcuje nazov LOCATION.
   !!! E. OPTION [--name SKU_TYPE] vo VALUE [SKU_TYPE] urcuje type STOCK KEEPING UNIT. Pre studijne ucely sa odporuca VALUE [Standard].
   F. OPTION [--min-tls TLS_VERSION] vo VALUE [TLS_VERSION] urcuje verziu TLS PROTOCOL. Odporuca sa VALUE [1.2].

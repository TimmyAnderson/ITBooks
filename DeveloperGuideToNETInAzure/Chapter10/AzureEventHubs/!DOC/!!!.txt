//----------------------------------------------------------------------------------------------------------------------
1. SOLUTION demonstruje pouzitie AZURE EVENT HUBS v .NET CORE.
2. AZURE EVENT HUBS je BIG DATA STREAMING PLATFORM a EVENT INGESTION SERVICE.
3. AZURE EVENT HUBS umoznuje spracovanie velkeho mnozstva dat, ktore mozu generovat rozlicne zariadenia, ako napriklad INTERNET OF THING DEVICES. Tieto DEVICES maju tendenciu generovat velke mnozstvo dat, ktore vyzaduje dodatocne spracovanie na casto velkom pocte MACHINES. AZURE EVENT HUBS umoznuje vyuzit AZURE PLATFORM na spracovanie tychto dat v REAL TIME.
!!! 4. AZURE EVENT HUBS je urceny na spracovanie velkeho mnozstva dat. Pre EVENT DRIVEN PROGRAMMING su vhodnejsie nasledujuce technologie.
   A. AZURE EVENT GRID. Pre AZURE EVENT GRID platia nasledujuce fakty.
      A. AZURE EVENT GRID je HIGH AVAILABILITY SERVICE urcena na spracovanie EVENTS, ktore si vymienaju jednotlive SERVICES.
      B. AZURE EVENT GRID umoznuje REACTIVE PROGRAMMING.
      C. AZURE EVENT GRID implementuje PUBLISHER-SUBSCRIBER MODEL.
      !!! D. AZURE EVENT GRID implementuje PUSH MODEL, ked EVENTS odoslane PUBLISHERS su OKAMZITE dorucene do SUBSCRIBERS.
   B. AZURE SERVICE BUS. Pre AZURE SERVICE BUS platia nasledujuce fakty.
      A. AZURE SERVICE BUS je ENTERPRISE MESSAGING PLATFORM podporujuca MESSAGE QUEUES, MESSAGE SESSIONS, MESSAGE TOPICS, ci MESSAGE SUBSCRIPTION.
      !!! B. AZURE SERVICE BUS implementuje PULL MODEL, kedy PUBLISHERS odosielaju MESSAGES do MESSAGE QUEUES, kde su ulozene, az kym sa SUBSCRIBERS nerozhodnu ich vycitat a spracovat.
5. AZURE EVENT HUBS spracuvaju EVENTS. Pre EVENTS platia nasledujuce fakty.
   A. EVENTS obsahuju casovu seriu dat, ktore su zaznamenane zdrojom dat, ako su napriklad INTERNET OF THING DEVICES, a ktoru si tymito DEVICES odosielane do AZURE EVENT HUBS prostrednictvom INTERNETU.
   !!!!! B. Klucovou charakteristikou EVENTS v AZURE EVENT HUBS, ze EVENTS prichadzaju KONTINUALNE vo forme STREAMU. Napriklad, ak su data merane INTERNET OF THING DEVICE kazdu minutu, tak prichadzaju kazdu minutu. Nejedna sa teda o jednorazove EVENTS, ako napriklad spustenie nejakeho procesu zo strany HUMAN USER na spracovanie ktorych su vhodnejsie technologie ako AZURE EVENT GRID a AZURE SERVICE BUS.
   C. EVENTS v AZURE EVENT HUBS obsahuju nasledujuce data.
      A. PAYLOAD. PAYLOAD su aktualne data, ktore DATA SOURCES generuju.
      B. METADATA. METADATA su popisne data, ktore popisuju PAYLOAD data.
6. EVENTS produkuju EVENT PRODUCERS. Pre EVENT PRODUCERS platia nasledujuce fakty.
   A. EVENT PRODUCER je DEVICE, alebo PROCESS, ktory generuje EVENTS.
   B. EVENT PRODUCERS odosielaju EVENTS prostrednictvom INTERNETU do AZURE EVENT HUBS, kde su tieto EVENTS docasne ulozene, aby mohli byt spracovane EVENT CONSUMERS.
   C. Prikladom EVENT PRODUCERS su INTERNET OF THING DEVICES, alebo LOG SYSTEMS, ktore generuju kontinualny STREAM EVENTS.
7. EVENTS spracuvaju EVENT CONSUMERS. Pre EVENT CONSUMERS platia nasledujuce fakty.
   A. EVENT CONSUMERS su PROCESSES, ktore su zodpovedne sa spracovanie EVENTS, ktore vyprodukovali EVENT PRODUCERS.
   B. EVENT CONSUMERS nacitavaju EVENTS z AZURE EVENT HUBS, kde su EVENTS docasne ulozene EVENT PRODUCERS a spracovavaju ich podla potreby.
   C. Prikladom EVENT CONSUMERS su SERVICES vykonavajuce spracovanie EVENTS, alebo DATABASES do ktorych sa EVENTS ukladane.
8. AZURE EVENT HUBS podporuju PARTITIONS. Pre PARTITIONS platia nasledujuce fakty.
   A. PARTITIONS umoznuju paralelne spracovanie EVENTS viacerymi EVENT CONSUMERS.
   B. Kazda PARTITION obsahuje vlastny CHAIN OF EVENTS, ktore drzia EVENTS po urcitu dobu, az kym nie su spracovane v EVENT CONSUMERS, alebo neuplynie TIMEOUT, kedy su EVENTS zahodene.
   !!! C. PARTITIONS reprezentuju nezavisle STREAMS EVENTS.
   D. Kazda PARTITION ma unikatny PARTITION IDENTIFIER.
   E. EVENTS je mozne odosielat do PARTITIONS nasledujucimi sposobmi.
      A. Pomocou PARTITION IDENTIFIER, kedy je EVENT odoslany do konkretnej PARTITION identifikovanej jej PARTITION IDENTIFIER.
      B. Pomocou PARTITION KEY, ktory reprezentuje HINT do ktorej PARTITION ma byt EVENT odoslany. Tento HINT nemusi AZURE EVENT HUBS respektovat a teda nie je garantovane, ze EVENTS s rovnakym PARTITION KEY skoncia v tej istej PARTITION.
9. AZURE EVENT HUBS podporuju EVENT HUBS NAMESPACES. Pre EVENT HUBS NAMESPACES platia nasledujuce fakty.
   A. EVENT HUBS NAMESPACES su CONTAINERS, ktore umoznuju spajanie viacerych EVENT HUBS do GROUPS.
   B. EVENT HUBS NAMESPACES zjednodusuju spravu EVENT HUBS.
10. EVENT INGESTION v AZURE EVENT HUBS prebieha nasledujucim sposobom.
   A. EVENTS su zasielane do AZURE EVENT HUBS pomocou PROTOCOL [AMQP], PROTOCOL [HTTPS] (HTTP POST) a PROTOCOL [APACHE KAFKA].
   B. EVENTS mozu byt zlucovane do BATCHES, pricom plati, ze maximalna dlzka BATCH je 1 MB.
   C. Po prichode EVENTS do AZURE EVENT HUBS su EVENTS zaradene do prislusnej AZURE EVENT HUBS PARTITION. V PARTITIONS su EVENTS ukladane podla casu v ktorom dorazili.
11. EVENT PROCESSING v AZURE EVENT HUBS prebieha nasledujucim sposobom.
   A. AZURE EVENT HUBS integruje viacero AZURE SERVICES, ktore mozu citat EVENTS z AZURE EVENT HUBS. Prikladom je AZURE EVENT GRID, AZURE FUNCTIONS, AZURE STREAM ANALYTICS, ci AZURE LOGIC APPS.
   B. EVENTS z AZURE EVENT HUBS mozu byt spracovavane aj pomocou CUSTOM PROGRAMS, ktore nebezia priamo v AZURE, ak vyuzivaju AZURE EVENT HUBS PROCESSOR PACKAGE.
12. EVENTS generovane EVENT PRODUCERS su v AZURE EVENT HUBS ukladane nasledujucim sposobom.
   A. EVENTS su ukladane do PARTITIONS na definovany cas, ktory sa nazyva RETENTION PERIOD.
   B. RETENTION PERIOD je BY-DEFAULT nastaveny na 1 den, ale da sa predlzit na 7 dni.
   C. EVENTS po prichode do PARTITIONS nie je mozne editovat ani vymazavat. Vymazavane su az po uplynuti RETENTION PERIOD.
   D. EVENTS su v PARTITIONS podla casu svojho prichodu.
   E. EVENTS su z PARTITIONS citane FIRST IN FIRST OUT (FIFO) sposobom.
   F. AZURE EVENT HUBS umoznuje pre potreby dlhej RETENTION PERIOD, REPLICATION, DISASTER RECOVERY a HIGH AVAILABILITY ukladat EVENTS do AZURE BLOB STORAGE a AZURE DATA LAKE STORAGE.
13. AZURE EVENT HUBS podporuje SCALING.
   A. SCALING v AZURE EVENT HUBS je realizovany pomocou THROUGHPUT UNITS.
   B. THROUGHPUT UNIT umoznuje spracovanie definovaneho poctu EVENTS za sekundu.
   C. Jedna THROUGHPUT UNIT garantuje spracovanie 1 MB EVENTS generovanych EVENT PRODUCERS a 2 MB EVENTS spracovanych v EVENT CONSUMERS.
   D. AZURE EVENT HUBS PARTITIONS umoznuju dalsiu uroven SCALING.
14. AZURE EVENT HUBS podporuje AVAILABILITY a RELIABLITY.
   A. AZURE EVENT HUBS automaticky replikuje PARTITIONS 3 krat v ramci toho isteho DATA CENTER.
   B. AZURE EVENT HUBS umoznuje nakonfigurovat replikaciu PARTITIONS do viacerych DATA CENTERS v ramci toho isteho AZURE REGION.
   C. AZURE EVENT HUBS umoznuje sparovat 2 AZURE EVENT HUBS NAMESPACES v roznych AZURE REGIONS. Pri vypadku jedneho AZURE REGION sa je mozne prepnut do AZURE EVENT HUBS NAMESPACE v sekundarnom AZURE REGION.
15. AZURE EVENT HUBS automaticky replikuje svoju INFRASTRUCTURE na zabezpecenie vysokej AVAILABILITY a RELIABLITY. Platia nasledujuce fakty.
   A. AZURE SERVICE [AVAILABILITY ZONES] automaticky distribuuje AZURE EVENT HUBS INFRASTRUCTURE medzi viacerymi DATA CENTERS v ramci toho isteho AZURE REGION na zvysenie AVAILABILITY a RELIABLITY.
   B. AZURE SERVICE [GEO DISASTER RECOVERY] vykonava parovanie PRIMARY AZURE EVENT HUBS NAMESPACE so SECONDARY AZURE EVENT HUBS NAMESPACE, ktory sa nechadza v inom AZURE REGION, vdaka comu sa da realizovat BACKUP na SECONDARY AZURE EVENT HUBS NAMESPACE v pripade vypadku PRIMARY AZURE REGION.
16. Vytvorenie EVENT HUBS HUB pozostava z nasledujucich krokov.
   A. Zaregistrovanie RESOURCE PROVIDER [Microsoft.EventHub].
   B. Vytvorenie EVENT HUBS NAMESPACE.
   C. Vytvorenie EVENT HUBS HUB.
17. S EVENT HUBS NAMESPACES sa pracuje pomocou nasledujucich COMMANDS.
   A. EVENT HUBS NAMESPACE sa vytvara pomocou COMMAND [az eventhubs namespace create]. Prikladom je COMMAND [az eventhubs namespace create --name MyNamespaceTA76 --resource-group Chapter10 --location centralus --sku Basic].
   B. EVENT HUBS NAMESPACE sa odstranuje pomocou COMMAND [az eventhubs namespace delete]. Prikladom je COMMAND [az eventhubs namespace delete --name MyNamespaceTA76 --resource-group Chapter10].
   C. Zoznam EVENT HUBS NAMESPACES vracia COMMAND [az eventhubs namespace list]. Prikladom je COMMAND [az eventhubs namespace list --resource-group Chapter10].
   D. EVENT HUBS NAMESPACE CONNECTION STRING sa ziskava pomocou COMMAND [az eventhubs namespace authorization-rule keys list]. Prikladom je COMMAND [az eventhubs namespace authorization-rule keys list --namespace-name MyNamespaceTA76 --resource-group Chapter10 --name RootManageSharedAccessKey].
18. S EVENT HUBS HUBS sa pracuje pomocou nasledujucich COMMANDS.
   A. EVENT HUBS HUB sa vytvara pomocou COMMAND [az eventhubs eventhub create]. Prikladom je COMMAND [az eventhubs eventhub create --name MyHubTA76 --namespace-name MyNamespaceTA76 --resource-group Chapter10 --retention-time 24 --cleanup-policy Delete].
   B. EVENT HUBS HUB sa odstranuje pomocou COMMAND [az eventhubs eventhub delete]. Prikladom je COMMAND [az eventhubs eventhub delete --name MyHubTA76 --namespace-name MyNamespaceTA76 --resource-group Chapter10].
   C. Zoznam EVENT HUBS HUBS vracia COMMAND [az eventhubs eventhub list]. Prikladom je COMMAND [az eventhubs eventhub list --namespace-name MyNamespaceTA76 --resource-group Chapter10].
19. EVENT HUBS HUBS podporuje koncept CHECKPOINTS. Pre CHECKPOINTS platia nasledujuce fakty.
   A. CHECKPOINT urcuje POSITION v EVENT HUBS PARTITION, z ktorej bol ako z poslednej uspesne nacitany EVENT v EVENT STREAM.
   B. CHECKPOINTS umoznuju EVENT CONSUMERS zaznamenat POSITION, kde skoncili so spracovanim EVENTS v EVENT HUBS PARTITIONS, aby nedochadzalo k duplicitnemu spracovanie tych istych EVENTS, alebo vynechaniu spracovania niektorych EVENTS.
   !!! C. CHECKPOINTS sa ukladaju do BLOB STORAGE. Preto EVENT CONSUMERS musia mat zaregistrovany BLOB STORAGE, aby mohli vycitavat EVENTS z EVENT HUBS PARTITIONS.
   !!! D. EVENT CONSUMERS mozu zapisovat CHECKPOINTS podla potreby. Plati, ze cim tazsie sa CHECKPOINT zapisuje, tym to ma vacsi dopad na vykonnost. Na druhej strane, nizsia frekvencia zapisov CHECKPOINTS moze viest k vzniku duplicitnych EVENTS.
20. .NET CLIENT LIBRARY pre AZURE EVENT HUBS sa nachadza v nasledujucich PACKAGES.
   A. .NET CLIENT LIBRARY pre AZURE EVENT HUBS PRODUCER je ulozena v PACKAGE [Azure.Messaging.EventHubs].
   B. .NET CLIENT LIBRARY pre AZURE EVENT HUBS CONSUMER je ulozena v PACKAGE [Azure.Messaging.EventHubs.Processor].
!!! 21. Pre odosielanie EVENTS do AZURE EVENT HUBS v EVENT HUBS PRODUCER je nutne vytvorit nasledujuce RESOURCES.
   A. EVENT HUBS NAMESPACE.
   B. EVENT HUBS HUB.
22. EVENTS su v AZURE EVENT HUBS odosielane nasledujucim sposobom.
   A. Vytvori sa instancia CLASS [EventHubProducerClient], ktora ma nasledujuce PARAMETERS.
      A. CONNECTION STRING na PARTITION. CONNECTION STRING na PARTITION sa da ziskat volanim COMMAND [az eventhubs namespace authorization-rule keys list].
      B. EVENT HUB NAME.
   B. Vytvoria sa instancie CLASS [EventData], ktore obsahuju EVENT DATA EVENTS.
   C. EVENTS sa odoslu do AZURE EVENT HUBS niektorym z nasledujucich sposobov.
      A. EVENTS sa odoslu ako COLLECTION EVENTS volanim METHOD [Task EventHubProducerClient.SendAsync(IEnumerable<EventData> EventBatch, CancellationToken CancellationToken)].
      B. EVENTS sa odoslu ako BATCH v nasledujucich krokoch.
         A. Vytvori sa instancia CLASS [EventDataBatch] volanim METHOD [ValueTask<EventDataBatch> EventHubProducerClient.CreateBatchAsync(CancellationToken CancellationToken)].
         B. EVENTS sa vlozia do BATCH volanim METHOD [bool EventDataBatch.TryAdd(EventData EventData)].
         C. Cely BATCH sa odosle do AZURE EVENT HUBS volanim METHOD [Task EventHubProducerClient.SendAsync(EventDataBatch EventBatch, CancellationToken CancellationToken)].
         D. Instancia CLASS [EventDataBatch] sa uvolni volanim METHOD [void EventDataBatch.Dispose()].
   D. Instancia CLASS [EventHubProducerClient] sa uvolni volanim METHOD [ValueTask EventHubProducerClient.DisposeAsync()].
23. CLASS [EventHubProducerClient] umoznuje odosielanie EVENTS do AZURE EVENT HUBS. Pre CLASS platia nasledujuce fakty.
   A. CLASS implementuje INTERFACE [IAsyncDisposable], ktory umoznuje asynchronne uvolnenie RESOURCES.
   B. CONSTRUCTOR vytvara instanciu CLASS [EventHubProducerClient]. Ako PARAMETER sa zadava CONNECTION STRING, ktory sa da ziskat pomocou COMMAND [az eventhubs namespace authorization-rule keys list].
   C. PROPERTY [string FullyQualifiedNamespace] obsahuje FULLY QUALIFIED EVENT HUBS NAMESPACE.
   D. PROPERTY [string EventHubName] obsahuje nazov EVENT HUBS HUB.
   E. PROPERTY [string Identifier] obsahuje unikatny identifikator AZURE EVENT HUBS PRODUCER.
   F. PROPERTY [bool IsClosed] obsahuje informaciu ci CONNECTION na AZURE EVENT HUBS je uzatvorena.
   G. METHOD [ValueTask DisposeAsync()] asynchronne uvolnuje RESOURCES AZURE EVENT HUBS CONNECTION.
   H. METHOD [Task CloseAsync(CancellationToken CancellationToken)] uzatvara CONNECTION na AZURE EVENT HUBS.
   I. METHOD [Task<PartitionPublishingProperties> GetPartitionPublishingPropertiesAsync(string PartitionId, CancellationToken CancellationToken)] vracia informacie o EVENT HUBS PARTITION do ktorej bude EVENT zaslany. METHOD vracia informacie o EVENT HUBS PARTITION az po odoslani aspon 1 EVENT.
   J. METHOD [Task<EventHubProperties> GetEventHubPropertiesAsync(CancellationToken CancellationToken)] vracia informacie o EVENT HUBS HUB do ktoreho bude EVENT zaslany. METHOD vracia informaciu o pocte PARTITIONS a PARTITION IDENTIFIERS.
   K. METHOD [Task<string[]> GetPartitionIdsAsync(CancellationToken CancellationToken)] vracia zoznam PARTITION IDENTIFIERS EVENT HUBS HUB ku ktoremu je OBJECT pripojeny.
   L. METHOD [Task<PartitionProperties> GetPartitionPropertiesAsync(string PartitionId, CancellationToken CancellationToken)] vracia informacie o PARTITION identifikovanej pomocou PARTITION IDENTIFIER EVENT HUBS HUB ku ktoremu je OBJECT pripojeny.
   M. METHOD [Task SendAsync(IEnumerable<EventData> EventBatch, SendEventOptions Options, CancellationToken CancellationToken)] odosiela mnozinu EVENTS do EVENT HUBS HUB.
   N. METHOD [ValueTask<EventDataBatch> CreateBatchAsync(CreateBatchOptions Options, CancellationToken CancellationToken)] vytvara EVENT BATCH pre odosielanie EVENTS do EVENT HUBS HUB.
   O. METHOD [Task SendAsync(EventDataBatch EventBatch, CancellationToken CancellationToken)] odosiela EVENTS ako BATCH do EVENT HUBS HUB.
24. CLASS [EventDataBatch] umoznuje odosielanie EVENTS do AZURE EVENT HUBS ako BATCH. Pre CLASS platia nasledujuce fakty.
   A. CLASS implementuje INTERFACE [IDisposable], ktory umoznuje uvolnenie RESOURCES.
   B. PROPERTY [long MaximumSizeInBytes] obsahuje maximalny pocet BYTES, ktore moze obsahovat BATCH.
   C. PROPERTY [long SizeInBytes] obsahuje pocet BYTES, ktore obsahuje BATCH.
   D. PROPERTY [int Count] obsahuje EVENTS, ktore obsahuje BATCH.
   E. METHOD [void Dispose()] uvolnuje RESOURCES AZURE EVENT BATCH.
   F. METHOD [bool TryAdd(EventData EventData)] sa pokusi vlozit EVENT do BATCH. METHOD vrati VALUE [true], ak sa EVENT podarilo vlozit do BATCH a VALUE [false], ak sa EVENT nepodarilo vlozit do BATCH, pretoze BATCH by prekrocil maximalnu dlzku.
25. CLASS [EventData] reprezentuje EVENT DATA daneho EVENT a EVENT METADATA daneho EVENT. Pre CLASS platia nasledujuce fakty.
   A. CLASS [EventData] je DERIVED z CLASS [MessageContent].
   B. CONSTRUCTOR vytvara instanciu CLASS [EventData].
   C. PROPERTY [BinaryData EventBody] obsahuje BINARY DATA asociovane s EVENT.
   D. PROPERTY [ReadOnlyMemory<byte> Body] obsahuje BINARY DATA asociovane s EVENT.
   E. PROPERTY [Stream BodyAsStream] obsahuje BINARY DATA asociovane s EVENT.
   F. PROPERTY [string ContentType] obsahuje MIME TYPE dat, ktore su asociovane s EVENT.
   G. PROPERTY [string MessageId] obsahuje CUSTOM IDENTIFIER daneho EVENT.
   H. PROPERTY [string CorrelationId] obsahuje CUSTOM CORRELATION IDENTIFIER daneho EVENT.
   I. PROPERTY [IDictionary<string,object> Properties] obsahuje METADATA daneho EVENT.
   J. PROPERTY [IReadOnlyDictionary<string,object> SystemProperties] obsahuje SYSTEM METADATA daneho EVENT.
   K. PROPERTY [long SequenceNumber] obsahuje SEQUENCE NUMBER daneho EVENT. SEQUENCE NUMBER urcuje poradove cislo s ktorym je EVENT vlozeny do EVENT HUBS PARTITION.
   L. PROPERTY [string OffsetString] obsahuje OFFSET s ktorym bol EVENT nacitany z danej EVENT HUBS PARTITION.
   M. PROPERTY [DateTimeOffset EnqueuedTime] obsahuje UTC TIME kedy bol EVENT vlozeny do PARTITION.
   N. PROPERTY [string PartitionKey] obsahuje PARTITION KEY daneho EVENT.
   O. METHOD [AmqpAnnotatedMessage GetRawAmqpMessage()] vracia reprezentaciu EVENT vo formate PROTOCOLU AMQP.

31. EVENTS su do PARTITIONS zaradzovane nasledujucim sposobom.
   A. BY-DEFAULT, AZURE EVENT HUBS zaradzuje EVENTS do jednotlivych PARTITIONS pomocou ROUND ROBIN sposobu, aby pocet EVENTS v jednotlivych PARTITIONS bol rovnomerny.
   B. EVENTS mozu byt zaslane do konkretnej PARTITION, ktora je identifikovana pomocou PARTITION IDENTIFIER. Platia nasledujuce fakty.
      A. PARTITION IDENTIFIER sa nastavuje pomocou PROPERTY [string SendEventOptions.PartitionId].
      B. EVENT HUBS HUBS garantuje, ze EVENT s nastavenym PARTITION IDENTIFIER bude odoslany do PARTITION s danym PARTITION IDENTIFIER.
      !!! C. PARTITION s nastavenym PARTITION IDENTIFIER musi existovat, aby do nej mohol AZURE EVENT HUBS odoslat EVENT.
   C. EVENTS mozu byt zaslane do PARTITION pomocou PARTITION KEY. Platia nasledujuce fakty.
      A. PARTITION KEY sa nastavuje pomocou PROPERTY [string SendEventOptions.PartitionKey].
      !!! B. EVENT HUBS HUBS NEGARANTUJE, do ktorej konkretne PARTITION bude EVENT s nastavenym PARTITION KEY realne odoslany. Je to iba HINT. To znamena, ze sa moze stat, ze EVENTS s rovnakym PARTITION KEY budu ulozene v rozdielnych PARTITIONS.
32. AZURE EVENT HUBS definuje CONSUMER GROUPS. Pre CONSUMER GROUPS platia nasledujuce fakty.
   A. CONSUMER GROUP reprezentuje VIEW EVENT CONSUMERS na EVENTS, ktore su ulozene v AZURE EVENT HUBS HUB.
   !!! B. Kazda CONSUMER GROUP obsahuje mnozinu OFFSETS (SEQUENCE NUMBERS) pre kazdu PARTITION v AZURE EVENT HUBS HUB, ktora je nezavisla na ostatnych CONSUMER GROUPS.
   C. CONSUMER GROUPS umoznuju EVENT CONSUMERS v CONSUMER GROUPS citanie a spracovanie EVENTS, ktore je NEZAVISLE na inych CONSUMER GROUPS.
   !!!!! D. Ak viacero EVENT CONSUMERS citat EVENTS z tej istej CONSUMER GROUP, AZURE EVENT HUBS rozdeli spracovanie EVENTS ulozenym v roznych PARTITIONS roznym EVENT CONSUMERS, aby EVENT CONSUMERS vzajomne necitali duplicitne tie iste EVENTS.
   !!! E. CONSUMER GROUPS umoznuju EVENT CONSUMERS zvolit nasledujuce strategie citania a spracovanie EVENTS.
      A. GREEDY PROCESSING. Pri GREEDY PROCESSING sa dany EVENT CONSUMER pokusa citat EVENTS z co najvacsieho poctu PARTITIONS.
      B. BALANCED PROCESSING. Pri BALANCED PROCESSING AZURE EVENT HUBS jednotlivym EVENT CONSUMERS priraduje rovnomerny pocet PARTITIONS, aby kazdy EVENT CONSUMER bol priblizne rovnako zatazeny.
   !!! F. CONSUMER GROUPS musia byt explicitne vytvorene pomocou COMMAND [az eventhubs eventhub consumer-group].
   G. Pri vytvoreni AZURE HUBS HUB sa vytvori DEFAULT CONSUMER GROUP s NAME [$Default].
   H. CONSUMER GROUPS SETTINGS sa nastavuje pomocou CLASS [EventProcessorClientOptions].
   !!! I. Pouzitie CONSUMER GROUPS vyzaduje STANDARD TIER.

AZURE STORAGE SERVICE ENCRYPTION vykonava ENCRYPTION ulozenych EVENTS pomocou 256 BITS AES ENCRYPTION. Tato ENCRYPTION je zapnuta BY-DEFAULT, pricom CUSTOMERS mozu definovat pouzitie vlastnych KEYS.

49. AZURE EVENT HUBS umoznuje definovat AUTO EXPIRATION POLICIES. AUTO EXPIRATION POLICIES umoznuju definovat cas ako dlho maju byt EVENTS ulozene v PARTITIONS predtym, nez su z nich odstranene.

APPLICATION REGISTRATIONS.

1. APPLICATION REGISTRATION je sposob akym zaregistrovat PROGRAM v AZURE, tak aby AZURE pre tento PROGRAM vygeneroval CLIENT ID a CLIENT SECRET, ktorymi PROGRAM v AZURE moze preukazat svoju IDENTITY.
2. APPLICATION REGISTRATION je mozne pouzit pre PROGRAMY, ktore bezia priamo v AZURE, ale aj pre PROGRAMY, ktore bezia mimo AZURE, vratane LOCAL MACHINES.
3. Overenie cez APPLICATION REGISTRATION neznamena, ze AZURE PLATROFM doveruje danemu PROGRAMU. Je to len dokaz, ze PROGRAM ma prislusne CREDENTIALS a na zaklade ACCESS RIGHTS, ktore su pridelene danemu APPLICATION REGISTRATION moze v AZURE vykonavat prislusne operacie.
4. APPLICATION REGISTRATION sa realizuje v nasledujucich krokoch.
   A. APPLICATION REGISTRATION sa vykona pomocou COMMAND [az ad app create].
   B. Po vytvoreni APPLICATION REGISTRATION je NUTNE vykonat PASSWORD RESET, aby sa vygeneroval CLIENT ID a CLIENT SECRET. PASSWORD RESET sa vykonava pomoocou COMMAND [az ad app credential reset].
   !!! C. Aby APPLICATION REGISTRATION bolo mozne pridelovat ROLES a ACCESS RIGHTS, je nutne pre APPLICATION REGISTRATION vytvorit SERVICE PRINCIPAL. SERVICE PRINCIPAL sa vytvara pomocou COMMAND [az ad sp create].
   D. Vytvorenemu SERVICE PRINCIPAL je nutne nastavit ROLES. ROLES sa nastavuju pomocou COMMAND [az role assignment create].
   !!! E. Pre AUTHENTICATION a AUTHORIZATION v AZURE je nutne CLIENT ID a CLIENT SECRET vlozit do instancie CLASS [ClientSecretCredential] a tu pouzit v AZURE API na AUTHENTICATION a AUTHORIZATION.

1. Solution demonstruje pouzitie GENERICS v JAVE.
2. V JAVE sa TYPE CHECKING vykonava na nasledujucich urovniach.
   A. TYPE CHECKING vykonany na urovni JAVA COMPILER. Pre TYPE CHECKING vykonanom na urovni JAVA COMPILER platia nasledujuce fakty.
      A. JAVA COMPILER vykonava COMPILE TIME TYPE CHECKING.
      B. JAVA COMPILER vykonava TYPE CHECKING na zaklade TYPE, ktora je priradena danej VARIABLE.
      !!! C. JAVA COMPILER TYPE CHECKING je mozne OBIST pomocou CASTING OPERATOR. V taktom pripade COMPILER nedokaze v COMPILE TIME skontrolovat ci TYPE danej VARIABLE zodpoveda TYPE daneho OBJECT, ktory VARIABLE obsahuje. To moze viest k vzniku RUN TIME ERRORS.
   B. TYPE CHECKING vykonany na urovni JAVA VIRTUAL MACHINE. Pre TYPE CHECKING vykonanom na urovni JAVA VIRTUAL MACHINE platia nasledujuce fakty.
      A. JAVA VIRTUAL MACHINE vykonava RUN TIME TYPE CHECKING.
      B. JAVA VIRTUAL MACHINE vykonava TYPE CHECKING na zaklade realneho TYPE, ktory dany OBJECT ma.
      C. Ak JAVA VIRTUAL MACHINE zisti, ze CODE skusa pre dany OBJECT zavolat MEMBER, ktory dany TYPE nema, pretoze v JAVA COMPILER bol pouzity CASTING OPERATOR, ktory obisiel JAVA COMPILER TYPE CHECKING, dojde k vzniku EXCEPTION [ClassCastException].
3. GENERICS umoznuje definovanie CODE CONSTRUCTS, ktore maju TYPE PARAMETERS. Pre GENERICS platia nasledujuce fakty.
   A. GENERICS su CODE CONSTRUCTS, ktore umoznuju definovat TYPE PARAMETERS, ktore nasledne mozu byt pouzite v CODE namiesto realnych TYPES, ktore sa substituuju, az je GENERICS CODE CONSTRUCT pouzity.
   B. GENERICS mozu byt aplikovane na nasledujuce CODE CONSTRUCTS.
      A. CLASSES.
      B. INTERFACES.
      C. RECORDS.
      D. METHODS.
   C. TYPE PARAMETERS sa definuju na zaciatku daneho CODE CONSTRUCT. Nasledne je mozne tieto TYPE PARAMETERS pouzivat v CODE daneho CODE CONSTRUCT.
   D. TYPE PARAMETERS su definovane pomocou SYNTAX [<TYPE_PARAMETER_DECLARATION_1,...,TYPE_PARAMETER_DECLARATION_N>], kde VALUES [TYPE_PARAMETER_DECLARATION_X] reprezentuju deklaraciu daneho TYPE PARAMETER.
   !!! E. Ako TYPE ARGUMENTS mozu byt pouzite iba OBJECTS, a nie PRIMITIVE TYPES.
   F. GENERICS TYPES mozu byt pouzite ako TYPE ARGUMENTS GENERICS TYPES.
   G. ARRAYS mozu byt pouzite ako TYPE ARGUMENTS do GENERICS TYPES.
   !!!!! H. JAVA NEPODPORUJE GENERICS ARRAYS. JAVA sice umozni deklaraciu GENERICS ARRAYS, ale pri pokuse o vytvorenie instancie GENERICS ARRAY, JAVA COMPILER hodi ERROR. Toto obmedzenie je dane obmedzenim BYTECODE v JAVA VIRTUAL MACHINE.
4. Pre GENERICS CLASSES platia nasledujuce fakty.
   A. V GENERICS CLASSES sa TYPE PARAMETERS definuju hned za nazvom CLASS pomocou SYNTAX [<TYPE_PARAMETER_DECLARATION_1,...,TYPE_PARAMETER_DECLARATION_N>], kde VALUES [TYPE_PARAMETER_DECLARATION_X] reprezentuju deklaraciu daneho TYPE PARAMETER.
   B. Pre GENERICS CLASSES sa TYPE ARGUMENTS definuju hned za nazvom CLASS pomocou SYNTAX [<TYPE_ARGUMENT_1,...,TYPE_ARGUMENT_N>], kde VALUES [TYPE_ARGUMENT_X] reprezentuju TYPE daneho TYPE ARGUMENT.
   C. V GENERICS CLASSES je mozne CLASS TYPE ARGUMENTS pouzit v CONSTRUCTOR a METHODS.
   !!! D. V GENERICS CLASSES NIE JE mozne pouzit CLASS TYPE ARGUMENTS v v STATIC METHODS.
5. Pre GENERICS INTERFACES platia nasledujuce fakty.
   A. V GENERICS INTERFACES sa TYPE PARAMETERS definuju hned za nazvom INTERFACE pomocou SYNTAX [<TYPE_PARAMETER_DECLARATION_1,...,TYPE_PARAMETER_DECLARATION_N>], kde VALUES [TYPE_PARAMETER_DECLARATION_X] reprezentuju deklaraciu daneho TYPE PARAMETER.
   B. Pre GENERICS INTERFACES sa TYPE ARGUMENTS definuju hned za nazvom INTERFACE pomocou SYNTAX [<TYPE_ARGUMENT_1,...,TYPE_ARGUMENT_N>], kde VALUES [TYPE_ARGUMENT_X] reprezentuju TYPE daneho TYPE ARGUMENT.
   C. TYPE, ktory implementuje GENERICS INTERFACE moze byt sam NON-GENERICS TYPE, alebo GENERICS TYPE. V pripade, ze sa jedna o GENERICS TYPE, tak GENERICS TYPE moze pouzit svoje TYPE PARAMETERS ako TYPE ARGUMENTS implementovanych GENERICS INTERFACES.
6. Pre GENERICS RECORDS platia nasledujuce fakty.
   A. V GENERICS RECORDS sa TYPE PARAMETERS definuju hned za nazvom RECORD pomocou SYNTAX [<TYPE_PARAMETER_DECLARATION_1,...,TYPE_PARAMETER_DECLARATION_N>], kde VALUES [TYPE_PARAMETER_DECLARATION_X] reprezentuju deklaraciu daneho TYPE PARAMETER.
   B. Pre GENERICS RECORDS sa TYPE ARGUMENTS definuju hned za nazvom RECORD pomocou SYNTAX [<TYPE_ARGUMENT_1,...,TYPE_ARGUMENT_N>], kde VALUES [TYPE_ARGUMENT_X] reprezentuju TYPE daneho TYPE ARGUMENT.
7. Pre GENERICS METHODS platia nasledujuce fakty.
   A. V GENERICS METHODS sa TYPE PARAMETERS definuju hned PRED RETURN VALUE TYPE pomocou SYNTAX [<TYPE_PARAMETER_DECLARATION_1,...,TYPE_PARAMETER_DECLARATION_N>], kde VALUES [TYPE_PARAMETER_DECLARATION_X] reprezentuju deklaraciu daneho TYPE PARAMETER.
   B. Pre GENERICS METHODS sa TYPE ARGUMENTS definuju EXPLICITNE hned PRED nazvom METHOD pomocou SYNTAX [<TYPE_ARGUMENT_1,...,TYPE_ARGUMENT_N>], kde VALUES [TYPE_ARGUMENT_X] reprezentuju TYPE daneho TYPE ARGUMENT.
   !!! C. Pri volani STATIC GENERICS METHODS s EXPLICITNE zadanymi TYPE ARGUMENTS je VZDY treba pouzit SYNTAX [CLASS_NAME.<TYPE_ARGUMENT_1,...,TYPE_ARGUMENT_N>.METHOD_NAME(ARGUMENT_1,...,ARGUMENT_N)], kde VALUE 
   !!! D. JAVA pre GENERICS METHODS dokaze vykonat TYPE INFERENCE, kedy su TYPE ARGUMENTS automaticky identifikovane z TYPES METHOD ARGUMENTS. V takomto pripade TYPE ARGUMENTS nie je nutne definovat EXPLICITNE.
   !!!!! E. Pri TYPE INFERENCE postupuje JAVA nasledovne.
      A. JAVA urci TYPES vsetkych METHOD ARGUMENTS.
      B. Ak ma METHOD 1 METHOD ARGUMENT, ktory ma TYPE z TYPE ARGUMENT, tak sa pouzije TYPE detekovany z TYPE METHOD ARGUMENT.
      C. Ak ma METHOD VIACERO METHOD ARGUMENTS, ktore maju ROVNAKY TYPE ARGUMENT, COMPILER postupuje nasledovne.
         A. Ak su TYPES vsetkych METHOD ARGUMENTS rovnake, tak sa pouzije tento TYPE.
         !!!!! B. Ak su TYPES METHOD ARGUMENTS ROZDIELNE, JAVA prezera zoznam BASE CLASSES a IMPLEMENTED INTERFACES a snazi sa najst rovnaky TYPE vsetkych METHOD ARGUMENTS. Kedze, kazdy OBJECT v JAVE dedi z CLASS [Object], tak v pripade, ze sa nenajde iny spolocny TYPE, pouzije sa TYPE [Object].
   F. GENERICS METHODS mozu byt definovane len za urcitych podmienok. Platia nasledujuce fakty.
      A. JAVA umoznuje definovat GENERICS CONSTRUCTORS v NON-GENERICS CLASSES.
      B. JAVA umoznuje definovat GENERICS CONSTRUCTORS v GENERICS CLASSES. Taketo CONSTRUCTORS mozu okrem METHOD TYPE ARGUMENTS pouzivat aj CLASS TYPE ARGUMENTS.
      C. JAVA umoznuje definovat NON STATIC GENERICS METHODS v NON-GENERICS CLASSES.
      D. JAVA umoznuje definovat NON STATIC GENERICS METHODS v GENERICS CLASSES. Taketo METHODS mozu okrem METHOD TYPE ARGUMENTS pouzivat aj CLASS TYPE ARGUMENTS.
      E. JAVA umoznuje definovat STATIC GENERICS METHODS v NON-GENERICS CLASSES.
      !!! F. JAVA umoznuje definovat STATIC GENERICS METHODS v GENERICS CLASSES, ale taketo METHODS mozu pouzivat IBA METHOD TYPE ARGUMENTS, ale NIE CLASS TYPE ARGUMENTS.
   G. Pri volani GENERICS METHODS je mozne vyuzit DIAMOND OPERATOR.
8. JAVA umoznuje zjednodusit pouzivanie GENERICS TYPES pouzitim DIAMOND OPERATOR. Pre DIAMOND OPERATOR platia nasledujuce fakty.
   A. DIAMOND OPERATOR umoznuje pri pouziti GENERICS TYPE nezadat TYPE ARGUMENTS, ak COMPILER dokaze z CODE CONTEXT jednoznacne urcit TYPE ARGUMENTS.
   B. DIAMOND OPERATOR ma SYNTAX [TYPE<>], kde VALUE [TYPE] je GENERICS TYPE.
   C. DIAMOND OPERATOR je mozne pouzit v nasledujucich CODE CONSTRUCTS.
      A. DIAMOND OPERATOR je mozne pouzit po OPERATOR NEW v INITIALIZATION GENERICS TYPE VARIABLE.
      B. DIAMOND OPERATOR je mozne pouzit po OPERATOR NEW mimo INITIALIZATION GENERICS TYPE VARIABLE.
      C. DIAMOND OPERATOR je mozne pouzit ak GENERICS TYPE je pouzity ako PARAMETER METHOD.
      D. DIAMOND OPERATOR je mozne pouzit ak GENERICS TYPE je pouzity ako RETURN TYPE METHOD.
   D. DIAMOND OPERATOR NIE JE mozne pouzit v nasledujucich CODE CONSTRUCTS.
      !!! A. DIAMOND OPERATOR NIE JE mozne pouzit v NESTED GENERICS TYPES pri definicie INTERNAL TYPE NESTED GENERICS TYPE. Ale DIAMOND OPERATOR je mozne pouzit v NESTED GENERICS TYPES, ak sa aplikuje na CELY NESTED GENERICS TYPE.
      !!! B. DIAMOND OPERATOR NIE JE mozne pouzit v OPERATOR NEW, ak za pouzitim OPERATOR nasleduje pouzitie dalsich METHODS vytvoreneho TYPE. Ale ak OPERATOR NEW pouziva parametericky CONSTRUCTOR, ktoreho PARAMETERS definuju GENERICS TYPES, potom DIAMOND OPERATOR je mozne pouzit.
9. JAVA definuje TYPE [Void]. Pre TYPE [Void] platia nasledujuce fakty.
   A. TYPE [Void] je CLASS TYPE, ktory nema ziadnu implementaciu.
   B. TYPE [Void] sa pouziva ako TYPE ARGUEMENT do GENERICS TYPES, ak METHODS GENERICS TYPES nemaju vracat ziadnu RETURN VALUE. V taktom pripade vsak tieto METHODS musia obsahovat CODE [return(null)].
10. JAVA pri preklade JAVA CODE do BYTE CODE vykonava pre GENERICS TYPES TYPE ERASURE. Pre TYPE ERASURE platia nasledujuce fakty.
   A. JAVA VIRTUAL MACHINE NEPODPORUJE GENERICS TYPES. Pre definovany GENERICS TYPE je na urovni JAVA VIRTUAL MACHINE je generovany iba 1 TYPE, a nie 'N' TYPES, osobitne pre kazdu kombinaciu TYPE ARGUMENTS.
   B. JAVA musi pri preklade JAVA CODE do JAVA VIRTUAL MACHINE BYTECODE vykonat TYPE ERASURE, kde z GENERICS TYPE nahradi TYPE PARAMETERS REPLACEMENT TYPES. REPLACEMENT TYPES mozu byt nasledujuci TYPES.
      A. Ak na TYPE PARAMETER NIE JE aplikovany BOUND TYPE, tak REPLACEMENT TYPE je TYPE [Object].
      B. Ak na TYPE PARAMETER NIE JE aplikovany BOUND TYPE, tak REPLACEMENT TYPE je BOUND TYPE.
   !!! C. V skutocnosti JAVA VIRTUAL MACHINE ma sice ulozenu informaciu o TYPE daneho TYPE ARGUEMENT v SIGNATURE ATTRIBUTES, ale realne ju nijako nevyuziva.
   D. JAVA COMPILER pri vykonava TYPE ERASUERE nielen v GENERICS CODE, ale aj na mieste, kde sa pouziva GENERICS CODE, pricom v pripade potreby do CODE automaticky vklada EXPLICIT CONVERSION TYPE PARAMETERS na TYPE ARGUMENTS.
11. TYPE ERASURE sposobuje mnozstvo obmedzeni pri pouziti GENERICS TYPE v JAVE.
   A. TYPE ERASURE znemoznuje vytvarat instancie TYPE PARAMETERS pomocou OPERATOR NEW, pretoze na urovni JAVA VIRTUAL MACHINE by to znamenalo vytvorit instanciu TYPE [Object].
   B. TYPE ERASURE znemoznuje v GENERICS CODE pouzitie OPERATOR [instanceof]. Je to preto, lebo na urovni JAVA VIRTUAL MACHINE existuje pre kazdy GENERICS TYPE iba 1 TYPE a tak by pouzitie OPERATOR [instanceof] pre GENERICS TYPE s roznymi TYPE PARAMETERS nedavalo zmysel.
   C. TYPE ERASURE znemoznuje v GENERICS CODE pouzitie CASTING na TYPE PARAMETER. Je to preto, lebo na urovni JAVA VIRTUAL MACHINE existuje pre kazdy GENERICS TYPE iba 1 TYPE, a tym padom by CASTING pre rozne TYPE PARAMETERS nemal zmysel.
   D. TYPE ERASURE znemoznuje v GENERICS CODE ziskanie CLASS TYPE pre GENERICS TYPES. Je to preto, lebo samotny CLASS TYPE [Class<TType>] je GENERICS TYPE, ktory podlieha TYPE ERASURE a teda na urovni JAVA VIRTUAL MACHINE existuje iba TYPE [Class]. Ale KEYWORD [class] je mozne aplikovat na GENERICS TYPE bez definovania TYPE PARAMETERS.
   E. TYPE ERASURE znemoznuje deklarovat GENERICS EXCEPTIONS. GENERICS EXCEPTION su CLASSES, ktore su DERIVED z CLASS [Throwable] a to JAVA NEUMOZNUJE. Je to preto, lebo po TYPE ERASURE na urovni JAVA VIRTUAL MACHINE by pre GENERICS EXCEPTION TYPE existoval iba 1 TYPE a tym padom nie je mozne definovat viacero CATCH CLAUSES podla GENERICS EXCEPTION TYPES, kedze by sa jednalo o CATCH CLAUSES na zaklade rovnakeho TYPE co JAVA VIRTUAL MACHINE neumoznuje.
   F. TYPE ERASURE znemoznuje deklarovat STATIC FIELDS v GENERICS TYPES. Je to preto, lebo STATIC FIELDS su asociovane s TYPE, a kedze na urovni JAVA VIRTUAL MACHINE existuje pre GENERICS TYPE iba 1 TYPE, nie je mozne definovat STATIC VARIABLES pre kazdu kombinaciu TYPE ARGUMENTS.
   G. TYPE ERASURE znemoznuje deklarovat STATIC METHODS v GENERICS TYPES. Je to preto, lebo na urovni JAVA VIRTUAL MACHINE existuje pre GENERICS TYPE iba 1 TYPE, a tym padom nie je mozne generovat STATIC GENERICS METHODS pre kazdu kombinaciu TYPE ARGUMENTS.
   H. TYPE ERASURE znemoznuje vykonavat METHOD OVERLOADING v GENERICS TYPES, ak OVERLOADED METHOD pouziva namiesto TYPE ARGUMENTS REPLACEMENT TYPES. Je to preto, lebo takato OVERLOADED METHOD by bola definovana v CODE GENERICS TYPE viacnasobne, co nie je mozne.
   I. TYPE ERASURE znemoznuje pouzit GENERICS TYPES pre METHOD OVERLOADING v NON-GENERICS TYPES. Je to preto, lebo po TYPE ERASURE existuje na urovni JAVA VIRTUAL MACHINE pre GENERICS TYPE iba 1 TYPE a tym padom METHOD OVERLOADING nie je mozny.
   J. TYPE ERASURE znemoznuje vytvarat ARRAYS GENERICS TYPES. Je to preto, lebo TYPE ERASURE znemoznuje pouzit taketo ARRAYS TYPE SEFE sposobom. JAVA vsak umoznuje vytvorenie ARRAYS pre GENERICS TYPE, ak sa pouzije RAW GENERICS TYPE bez definovanych TYPE ARGUMENTS.
12. GENERICS TYPES, ktore nemaju definovany TYPE ARGUMENTS sa nazyvaju RAW TYPES. Pre RAW TYPES platia nasledujuce fakty.
   A. RAW TYPE je GENERICS TYPE, ktory nema definovane TYPE ARGUMENTS.
   !!! B. Pouzitie RAW TYPES v JAVE je legalne, kedze v JAVA VIRTUAL MACHINE sa aj tak pouzivaju iba RAW TYPES.
   !!! C. RAW TYPES sa neodporuca pouzivat, pretoze dochadza k strate typovej informacie a TYPE PARAMETERS su nahradene REPLACEMENT TYPES.
   !!! D. Pri pouziti RAW TYPES COMPILER hodi WARNING.
   !!! E. RAW TYPES su implicitne konverovane na GENERICS TYPES s definovanymi TYPE ARGUMENTS. To moze viest k zlej konverzii na iny TYPE a preto COMPILER vzdy pri takejto CONVERSION hodi WARNING.
13. JAVA umoznuje v GENERICS METHODS CASTING na TYPE PARAMETERS. Ale COMPILER pre taketo CONVERSIONS hodi WARNING. WARNING je mozne potlacit aplikovanim ANNOTATION [@SuppressWarnings("unchecked")] na GENERICS METHOD.
14. JAVA umoznuje definovat TYPE RESTRICTIONS na TYPE PARAMETERS pomocou BOUNDS. Pre BOUNDS platia nasledujuce fakty.
   A. BOUND umoznuje pre kazdy TYPE PARAMETER definovat BASE CLASS a zoznam INTERFACES, ktore musi TYPE ARGUMENT splnat, aby mohol byt pouzity pre dany TYPE PARAMETER.
   B. BOUND ma SYNTAX [TYPE_PARAMETER extends CLASS_TYPE & INTERFACE_TYPE_1 & ... & INTERFACE_TYPE_N], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER, VALUE [CLASS_TYPE] je nazov BASE CLASS, z ktorej musi TYPE PARAMETER DERIVED a VALUES [INTERFACE_TYPE_X] su nazvy INTERFACE TYPES, ktore musi TYPE PARAMETER implementovat.
   C. JAVA umoznuje pouzit GENERICS CLASSES a GENERICS INTERFACES ako BOUNDS TYPE PARAMETERS.
   !!! D. JAVA umoznuje pouzit GENERICS PARAMETERS v BOUNDS. Ak sa pouzije GENERICS PARAMETER v BOUND, tak sa jedna o RECURSIVE TYPE BOUND.
   !!! E. Ak ma TYPE PARAMETER definovany BOUND, GENERICS CODE moze pre dany TYPE PARAMETER pouzit vsetky METHODS BASE CLASS a METHODS INTERFACE TYPES, ktore su definovane danym BOUND.
15. JAVA umoznuje pouzivat TYPE PARAMETERS v THROWS CLAUSE. Pre pouzitie TYPE PARAMETERS v THROWS CLAUSE platia nasledujuce fakty.
   A. Aby TYPE PARAMETER mohol byt pouzity v THROWS CLAUSE, musi byt pre TYPE PARAMETER definovany BOUND na BASE TYPE [Throwable].
   B. Ak sa ako BOUND pouzije BASE TYPE [Throwable], tak COMPILER akceptuje CHECKED aj UNCHECKED EXCEPTIONS.
   C. Ak sa ako BOUND pouzije BASE TYPE [RuntimeException], tak COMPILER akceptuje iba UNCHECKED EXCEPTIONS.
16. Pre COVARIANCE a INVARIANCE GENERIC TYPES platia nasledujuce fakty.
   !!! A. GENERIC TYPES su INVARIANT. To znamena, ze konverzia GENERIC TYPE, ktory ma ako TYPE PARAMETER DERIVED TYPE na GENERIC TYPE, ktory ma ako TYPE PARAMETER BASE TYPE NIE JE MOZNA.
   !!! B. ARRAYS su vzdy COVARIANT. To znamena, ze konverzia ARRAY TYPE pre DERIVED TYPE na ARRAY TYPE pre BASE TYPE je mozna.
17. JAVA umoznuje pre TYPE ARGUMENTS pouzit WILDCARD TYPES. Pre WILDCARD TYPES platia nasledujuce fakty.
   A. WILDCARD TYPE je TYPE ARGUMENTS, ktory je pouzity pri definovani GENERICS TYPE.
   B. WILDCARD TYPE pouziva SYNTAX [?] na oznacenie TYPE ARGUMENT.
   !!! C. WILDCARD TYPE znamena, ze TYPE ARGUMENT daneho TYPE PARAMETER v GENERICS TYPE je NEDEFINOVANY. To znamena, ze COMPILER nema ziadnu informaciu o tom, akeho TYPE je TYPE ARGUMENT.
   !!! D. WILDCARD TYPES sa pouzivaju vtedy, ak je potrebne pracovat s GENERICS TYPES, ale NIE je potrebny pristup k FIELDS ci METHODS, ktore pouzivaju dany TYPE PARAMETER. Prikladom moze byt vratenie poctu ITEMS v TYPE [List<TType>], kde METHOD [int List<TType>.size()] nepouziva TYPE PARAMETER.
   E. WILDCARD TYPES sa aplikuju iba v GENERICS METHODS. Nie v GENERICS TYPES.
   !!! F. WILDCARD TYPES rozlisuju METHODS nasledujucim sposobom.
      A. READ METHODS. Pre READ METHODS platia nasledujuce fakty.
         A. READ METHOD je METHOD, ktora ma ako RETURN VALUE TYPE TYPE PARAMETER, ktory bol nahradeny WILDCARD TYPE.
         !!!!! B. WILDCARD TYPES UMOZNUJU pouzit READ METHODS, kde RETURN VALUE TYPE je nahradeny TYPE [Object], pretoze COMPILER nema informaciu o skutocnom TYPE, ktory reprezentuje WILDCARD TYPE.
      B. WRITE METHODS. Pre WRITE METHODS platia nasledujuce fakty.
         A. WRITE METHOD je METHOD, ktora ma ako METHOD PARAMETER TYPES TYPE PARAMETERS, ktore boli nahradeny WILDCARD TYPE.
         !!!!! B. WILDCARD TYPES NEUMOZNUJU pouzit WRITE METHODS, pretoze METHOD PARAMETERS, ktorych TYPES boli nahradene WILDCARD TYPE mozu mat lubovolny TYPE. Ak by pouzitie WRITE METHODS COMPILER umoznil, doslo by k naruseniu TYPE SAFETY.
   G. WILDCARD TYPES umoznuju pouzitie BOUNDS. Pre pouzitie BOUNDS platia nasledujuce fakty.
      A. BOUNDS aplikovane na WILDCARD TYPES umoznuju obmedzit, ktore TYPES mozu byt pouzite ako TYPE ARGUMENTS pre TYPE PARAMETER, ktory je nahradeny WILDCARD TYPE.
      B. WILDCARD TYPES umoznuju definovat nasledujuce typy BOUNDS.
         A. UPPER BOUNDED WILDCARD TYPES. Pre UPPER BOUNDED WILDCARD TYPES platia nasledujuce fakty.
            A. UPPER BOUNDED WILDCARD TYPES definuju TYPES, ktore musi dany TYPE PARAMETER implementovat, aby bol akceptovany ako TYPE ARGUMENT.
            !!! B. UPPER BOUNDED WILDCARD TYPES urcuju, ze za TYPE ARGUMENTS su akceptovane tie TYPES, ktore implementuju dany BOUND TYPE. BOUND TYPE moze byt BASE CLASS i INTERFACE.
            C. UPPER BOUNDED WILDCARD TYPE ma SYNTAX [? extends TYPE], kde VALUE [TYPE] urcuje TYPE, ktory musi TYPE PARAMETER implementovat aby mohol byt pouzity ako TYPE ARGUMENT.
            !!! D. UPPER BOUNDED WILDCARD TYPE moze definvat iba 1 TYPE. JAVA NEPODPORUJE definiciu viacerych TYPES, ako je to u BOUNDS pre TYPE PARAMETERS.
            !!! E. Pri aplikovani UPPER BOUNDED WILDCARD TYPE platia pre READ METHODS nasledujuce fakty.
               A. Pri aplikovani UPPER BOUNDED WILDCARD TYPE je mozne volanie READ METHODS.
               B. Pri volani READ METHODS, RETURN VALUE TYPE je nahradeny BOUND TYPE, pretoze COMPILER vie, ze TYPE PARAMETER musi implementovat BOUND TYPE.
               C. Pri volani READ METHOD je METHOD [TYPE_PARAMETER MyMethod()], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER volana ako METHOD [BOUND_TYPE MyMethod()], kde VALUE [BOUND_TYPE] je BOUND TYPE aplikovany na TYPE PARAMETER.
            !!! F. Pri aplikovani UPPER BOUNDED WILDCARD TYPE platia pre WRITE METHODS nasledujuce fakty.
               A. Pri aplikovani UPPER BOUNDED WILDCARD TYPE NIE je mozne volanie WRITE METHODS.
               B. Pouzitie WRITE METHODS by znamenalo, ze METHOD [void MyMethod(TYPE_PARAMETER Parameter)], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER by sa mala volat ako METHOD [void MyMethod(BOUND_TYPE Parameter)], kde VALUE [BOUND_TYPE] je BOUND TYPE aplikovany na TYPE PARAMETER.
               C. Taketo volanie by sposobilo narusenie TYPE SAFETY, pretoze TYPE_PARAMETER moze byt LUBOVOLNY TYPE, ktory implementuje dany BOUND TYPE, a nie je ho mozne nahradit pouzitim BOUND TYPE. Volanie WRITE METHOD by viedlo k naruseniu TYPE SAFETY.
         B. LOWER BOUNDED WILDCARD TYPES. Pre LOWER BOUNDED WILDCARD TYPES platia nasledujuce fakty.
            A. LOWER BOUNDED WILDCARD TYPES definuju TYPES, z ktoreho musi by dany TYPE PARAMETER DERIVED, aby bol akceptovany ako TYPE ARGUMENT.
            !!! B. LOWER BOUNDED WILDCARD TYPE urcuje, ze ako TYPE ARGUMENT moze byt pouzity dany BOUND TYPE, ako aj vsetky BASE TYPES, ktore dany BOUND TYPE implementuje. BOUND TYPE moze byt BASE CLASS i INTERFACE.
            C. LOWER BOUNDED WILDCARD TYPE ma SYNTAX [? super TYPE], kde VALUE [TYPE] urcuje TYPE a vsetky jeho BASE CLASSES, ktore mozu byt pouzite ako TYPE ARGUMENTS.
            !!!!! D. Pri aplikovani LOWER BOUNDED WILDCARD TYPE platia pre READ METHODS nasledujuce fakty.
               A. Pri aplikovani LOWER BOUNDED WILDCARD TYPE je mozne volanie READ METHODS.
               B. Pri volani READ METHODS, RETURN VALUE TYPE je nahradeny TYPE [Object], pretoze COMPILER nevie akym REALNYM TYPE bol TYPE PARAMETER nahradeny.
               C. Pri volani READ METHOD je METHOD [TYPE_PARAMETER MyMethod()], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER je volana ako METHOD [Object MyMethod()].
            !!!!! E. Pri aplikovani LOWER BOUNDED WILDCARD TYPE platia pre WRITE METHODS nasledujuce fakty.
               !!!!! A. Pri aplikovani LOWER BOUNDED WILDCARD TYPE je mozne volanie WRITE METHODS, iba ak ako PARAMETER TYPES pouzivaju BOUND TYPE, ktory bol aplikovany na dany TYPE PARAMETER.
               !!! B. Pri aplikovani LOWER BOUNDED WILDCARD TYPE je volanie METHOD [void MyMethod(TYPE_PARAMETER Parameter)], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER nahradene volanim METHOD [void MyMethod(BOUND_TYPE Parameter)], kde VALUE [BOUND_TYPE] je BOUND TYPE aplikovany na TYPE PARAMETER. Taketo volanie je mozne, pretoze TYPE PARAMETER moze byt nahradeny za BOUND TYPE, alebo za BASE CLASSES BOUND TYPE a BOUND TYPE je vzdy konvertovatelny na BASE CLASSES BOUND TYPE.
               !!!!! C. Pri aplikovani LOWER BOUNDED WILDCARD TYPE NIE je mozne volanie WRITE METHODS, ak ako PARAMETER TYPES pouzivaju BASE CLASSES daneho BOUND TYPE, ktory bol aplikovany na dany TYPE PARAMETER.
               !!!!! D. Pri aplikovani LOWER BOUNDED WILDCARD TYPE by volanie METHOD [void MyMethod(TYPE_PARAMETER Parameter)], kde VALUE [TYPE_PARAMETER] je nazov TYPE PARAMETER bolo nahradene volanim METHOD [void MyMethod(BASE_BOUND_TYPE Parameter)], kde VALUE [BASE_BOUND_TYPE] je BASE CLASS BOUND TYPE, ktory je aplikovany na TYPE PARAMETER. Taketo volanie je mozne, pretoze TYPE PARAMETER moze byt nahradeny za BOUND TYPE, alebo za BASE CLASSES BOUND TYPE a BOUND TYPE je vzdy konvertovatelny na BASE CLASSES BOUND TYPE. To by narusilo TYPE SAFETY, pretoze TYPE PARAMETER moze byt aj samotny BOUND TYPE.
18. Pri aplikacii LOWER BOUNDS a UPPER BOUNDS na WILDCARD TYPES sa odporuca pouzitie PRODUCER EXTENDS CONSUMER SUPER princip. Pre tento princip platia nasledujuce fakty.
   A. Ak METHOD PARAMETER WILDCARD TYPE vola READ METHODS, teda jedna sa o PRODUCER dat, tak na WILDCARD TYPE je potrebne aplikovat UPPER BOUND. Teda pouzije sa KEYWORD [extends].
   B. Ak METHOD PARAMETER WILDCARD TYPE vola WRITE METHODS, teda jedna sa o CONSUMER dat, tak na WILDCARD TYPE je potrebne aplikovat LOWER BOUND. Teda pouzije sa KEYWORD [super].
19. CLASS [Class<TType>] je oznacovana ako TYPE TOKEN. Pre TYPE TOKENS platia nasledujuce fakty.
   A. TYPE TOKEN obsahuje informacie o danom TYPE.
   B. TYPE TOKEN sa da ziskat nasledujucimi sposobmi.
      A. Pomocou KEYWORD [class] aplikovanym na dany TYPE.
      !!! B. Pomocou METHOD [Class<?> Object.getClass()]. Tato METHOD je NON GENERICS a preto vracia RETURN VALUE TYPE [Class<?>]. Tento TYPE je vsak mozne IMPLICITNE pretypovat na TYPE [Class<? extends TYPE>], kde VALUE [TYPE] je TYPE daneho OBJECT.
   !!! C. CLASS [Class<TType>] NEDOKAZE reprezentovat GENERICS TYPES. TYPE PARAMETER [TType] nemoze byt sam GENERICS TYPE. Pre GENERICS TYPES je vsak mozne ziskat ich TYPE TOKEN nasledujucim sposobom.
      A. Definuje sa GENERICS ABSTRACT CLASS, ktora ma ako TYPE PARAMETER GENERICS TYPE.
      B. V GENERICS HELPER CLASS sa definuje METHOD, ktora vracia instanciu CLASS [Type], ktora obsahuje TYPE INFORMATION o GENERICS TYPE. Tato METHOD vykonava nasledujuce kroky.
         A. Zavola METHOD [Class<TType> Object.getClass()], ktora vrati [Class<TType>] pre GENERICS HELPER CLASS.
         B. Pre instanciu CLASS [Class<TType>] zavola METHOD [Type Class<TType>.getGenericSuperclass()], ktora vrati GENERICS SUPERCLASS. Vratenu RETURN VALUE pretypuje na TYPE [ParameterizedType].
         C. Pre instanciu CLASS [ParameterizedType] sa zavola METHOD [Type[] ParameterizedType.getActualTypeArguments()], ktora vrati zoznam TYPE ARGUMENTS.
         D. METHOD vrati prvy TYPE ARGUMENT ako RETURN VALUE.
      C. Pre GENERICS ABSTRACT CLASS sa pre kazdy GENERICS TYPE, ktoreho TYPE ma byt INFERRED sa definuje DERIVED CLASS, ktora implementuje GENERICS ABSTRACT CLASS pricom ako TYPE PARAMETER pouzije prislusny TYPE PARAMETER, ktorho TYPE ma byt INFERRED.
      D. Zavola sa METHOD DERIVED CLASS na ziskanie TYPE INFORMATION o GENERICS TYPE.
20. Aj ked v GENERICS CODE nie je mozne vytvarat instancie TYPE PARAMETERS pomocou OPERATOR [OPERATOR new], instancie TYPE PARAMETERS je mozne vytvarat pomocou CLASS [Class<TType>]. Platia nasledujuce fakty.
   A. Do GENERIC CODE sa ako PARAMTER zasle instancia CLASS [Class<TType>], kde VALUE [TType] je nazov TYPE PARAMETER. CLASS [Class<TType>] sa nazyva aj TYPE TOKEN.
   B. Pre instanciu CLASS [Class<TType>] sa zavola METHOD [Constructor<TType> Class<TType>.getConstructor(Class<?>... parameterTypes)] ktora vrati prislusny CONSTRUCTOR TYPE PARAMETER [TType].
   C. Pre instanciu CLASS [Constructor<TType>] sa zavola METHOD [public TType newInstance(Object... initargs)], ktora vrati instanciu TYPE PARAMETER [TType].
21. JAVA NEUMOZNUJE vytvorit nove instancie ARRAYS pre TYPE PARAMETERS pomocou OPERATOR [OPERATOR new]. Jedna sa o dosledok TYPE ERASURE. ARRAYS pre TYPE PARAMETERS je mozne vytvorit pomocou REFLECTION nasledujucim sposobom.
   A. Do GENERICS CODE je nutne zaslat instanciu CLASS [Class<TType>] pre ktoru ma byt ARRAY pre TYPE PARAMETER vytvoreny.
   B. Volanim METHOD [Object Array.newInstance(Class<?> componentType, int length)] je mozne vytvorit ARRAY pre TYPE PARAMETER. ARRAY musi byt pretypovany na ARRAY daneho TYPE PARAMETER.
